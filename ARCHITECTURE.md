# Codex — Architecture & Design Specification

> Internal codename: **codex** (repository, package names `@codex/*`, Tauri bundle identifier, technical documentation, this file)
> Product surface name: **Starphire** (user-facing throughout the UI — window title, splash, sign-in, all on-screen copy)
> Company: **Starphire**
> Status: Pre-V1 specification

**Product surface naming.** Codex is the internal / repository name and stays in technical documentation, code, package names (`@codex/desktop`, `@codex/shared`, `@codex/parsers`), Cargo crate (`flashback` is the legacy crate name and is *not* user-visible — see § 12.3 for the V1.5 rename follow-up), and the Tauri bundle identifier `com.codex.app`. **Starphire** is the product as the user encounters it: the word in the window title, in the sign-in wordmark, in `<title>`, and wherever the app introduces itself. **Starphire** is also the company; app name and company name are intentionally identical, so the splash footer publication mark and the wordmark above it both read "Starphire" — set in different typographic registers (Fraunces display vs. IBM Plex Mono uppercase 10px) so they read as separate layers rather than as duplicate logotype. The historical product name *Flashback* (used in the masthead through rev 49) and the brief interim name *Rune* (Day 1 working title before the cool/clinical aesthetic redirect) are both retired in favor of *Starphire* as of the Day 1 design sprint. Likewise the brief interim company name *Elendil* (which appeared in the Day 1 splash footer before the aesthetic redirect) is retired. Deep-link scheme `flashback://` and the Tauri Cargo crate name `flashback` both predate every rename; both are V1.5 follow-ups since changing them requires a Tauri rebuild and a deep-link migration plan, neither of which is blocking the user-facing rename.

- 2026-05-12 (rev 49): M16 closing finding — blob URLs must be created and revoked inside the same `useEffect`. **Section 12.6** gains a new entry "Blob URLs created and revoked inside the same `useEffect` for StrictMode safety" appended to the M14 Phase 3 desktop cluster (parallel structure to the rev 40 *Blob URL transport* entry it supersedes the implementation of). *The finding:* `apps/desktop/src/hooks/useEncryptedBlob.ts` originally created the URL via `useMemo` and revoked it in a paired `useEffect` cleanup keyed on `[url]`. Under React 18 `<StrictMode>` in development the pattern is broken on initial mount by construction: (a) StrictMode runs the `useMemo` factory twice during the synthetic-double-render purity check, so each fresh mount creates **two** `URL.createObjectURL` registrations — the first is orphaned forever (no cleanup ever sees it), the second becomes the committed `url` state; (b) StrictMode then runs effect setup → simulated unmount → simulated remount synchronously after commit, and the simulated unmount's cleanup invokes `URL.revokeObjectURL` on the committed URL **before** the browser finishes loading from `<img src>`. The browser then 404s the dead `blob:` URL with `net::ERR_FILE_NOT_FOUND` and renders a broken-image icon. Every photo, on every fresh mount. *Why it was latent:* the defect predates this rev — the `useMemo`+`useEffect` pattern shipped in M14 Phase 3 (commit `9d00e52`) and `<StrictMode>` has wrapped the React tree since project scaffold (`bc58bad`). Phase 3 verified successfully because the founder's archive at that point was ~1.3k photos in a small visible window, and HMR cycles during dev verification masked the synthetic-unmount path enough times to look like it was working. M15 closing's rev 47 refactor of `timeline.tsx` (queryFn-decrypts instead of downstream-useEffect) shortened the render cycle that first mounts `AndroidGalleryPhotoCard`, removing the timing slack that had been hiding the bug — every photo broke consistently from then on. The user surfaced the regression while verifying M16's Maps Timeline implementation; the four-question diagnostic in this rev's commit cluster localized it (StrictMode + the split `useMemo`/`useEffect` pattern is the root cause; render-cycle timing made it consistently visible). *Fix:* refactor to `useState` + a single `useEffect` that calls `URL.createObjectURL` AND `URL.revokeObjectURL` against a closure-captured `objectUrl` local. Each effect invocation owns its own URL — the synthetic-unmount path revokes the first invocation's URL cleanly, the synthetic-remount path creates a fresh URL that is still alive when React batches the two `setUrl` calls and commits the final render. No orphans, no premature revoke, no leaks. *Generalization codified in the new § 12.6 entry:* **any hook creating a browser-resource handle (object URLs, audio contexts, `MediaStream`s, `RTCPeerConnection`s, `IntersectionObserver`s, etc.) must use the in-same-effect pattern.** The `useMemo`-creates + `useEffect`-revokes split is a known anti-pattern under React 18 StrictMode and silently broke `useEncryptedBlob` from M14 Phase 3 through M15+M16 before being caught. The fix lands as a code commit in this rev's cluster (one file: `apps/desktop/src/hooks/useEncryptedBlob.ts`; ~15 lines of net change; the hook's external return contract is unchanged, so no callers are touched). - 2026-05-10 (rev 31): M14 phase 1 streaming AES-GCM honesty pass. Investigation during M14 phase 1 (cross-platform blob encryption test vectors) surfaced that the Rust `aes-gcm` crate (0.10) does not expose an incremental update / finalize API for single-shot streaming AES-GCM — the maintainers explicitly declined to add one (RustCrypto/AEADs#556) on the grounds that buffering complicates one-shot performance optimization. The `aes-gcm` crate's `aead::stream` module implements the STREAM construction (per-chunk nonces with per-chunk tags), which is a different wire format from what Section 6's blob encryption pattern calls for. Three viable paths existed: (α) add `aes-gcm-stream` as a new dependency exposing single-nonce + trailing-tag streaming, (β) keep `aes-gcm` 0.10 and back the chunked API surface with single-shot encryption that buffers plaintext internally, (δ) reimplement GCM streaming manually atop the `aes` and `ghash` crates. Decision: option β. Reasoning: thumbnail-first ingestion in M14 produces blobs in the 150KB–5MB range where memory pressure during single-shot encryption is negligible; the wire format (ciphertext bytes ‖ 16-byte tag, with nonce stored separately on `blob_ref`) is byte-identical to a future true-streaming implementation, so the upgrade path is forward-compatible without re-encrypting existing blobs; zero new crypto-adjacent dependencies in a privacy-first product; Kotlin parity is unambiguous because JDK `Cipher`'s `update()` / `doFinal()` for `AES/GCM/NoPadding` produces the same bytes as a single-shot encrypt on the accumulated buffer. Section 6's blob storage subsection updated to reflect the V1 implementation honestly — the chunk_size parameter affects API buffering only, not output bytes, and full plaintext is held in memory during encrypt/decrypt in V1. ## 0. Project Direction

This document describes Codex/Flashback as a **personal data intelligence layer**. The product was originally framed as a "private personal archive" (life-journal direction). As of rev 26, the direction has shifted toward a **productivity-and-reflection power tool**: a unified intelligence layer over multi-source personal data that reasons across sources, generates useful artifacts (briefings, reports, digests), and supports selective public sharing. Sections 1–3 reflect the current direction; revision history captures the shift in rev 26. This document is the source of truth — when in doubt, reference the latest revision.

---

## 1. Product Summary

Codex (product name "Flashback", subject to change) is a **personal data intelligence layer**. It ingests a person's scattered digital footprint across platforms — messages, emails, calendar, location, listens, photos, documents, manual notes — and provides three primary capabilities:

1. **Reason across all of it.** Ask natural-language questions, get grounded answers with citations to source events. Generate insights ("your driving patterns vs. your listening habits"), reports (Felton-style monthly recaps), and briefings ("everything I've exchanged with Jordan about the Acme deal").
2. **Generate useful artifacts.** One-prompt creations that compile cross-platform context — a collage from Sarah's birthday, an expense summary for tax season, a slide-deck compilation for the marketing team last quarter, a highlight reel of YouTube watched while commuting.
3. **Share curated views publicly.** Toggle individual events or grouped digests to public access — a freelancer's Q1 client digest, a personal year-in-review, a project postmortem — with backlinks to originals where possible. Selective, not all-or-nothing.

The product is positioned as a **power tool for productivity and reflection**, not a feelings-based journal. It serves freelancers, consultants, power users, small teams, and anyone who lives in multiple apps and wants their fragmented context unified.

**Core promise to the user:** *Your fragmented digital life, reasoned over and made useful — with the parts you choose, shared the way you want.*

**The product IS:**
- A unified intelligence layer over multi-source personal data
- A search and reasoning surface for one's complete digital context
- A generative tool that produces artifacts, briefings, reports, and digests
- A privacy-first product where the user controls what's private vs. shared
- An emotionally weighty product handling deeply sensitive data
- A consumer-feeling product (warm, visually rich, shareable) that power users also love (deep features, keyboard nav, advanced commands available but not foregrounded). Apple's playbook, not Linear's.

**The product is NOT:**
- A social network or feed
- A consumption dashboard for unread content
- A backup product for active platform data
- A "life journal" or feelings-tracker
- A tool that requires manual logging — most data is ingested automatically

This framing affects every design decision. When in doubt, ask: *does this make the user's fragmented digital context more useful, more reasoned-over, or more shareable on their terms?* If not, defer or cut.

## 2. Foundational Principles

These are non-negotiable. All other decisions should be consistent with these.

1. **Privacy is the product.** Sensitive data is encrypted on the client before it reaches our servers. We cannot read the user's archive content. This is not a marketing claim; it is a structural property of the system. Privacy applies to *private* data — publicly shared entries are by definition not private; the user explicitly opts each entry into sharing (see Section 7's "Public sharing and the encryption boundary").
2. **The user owns their data.** Account deletion is a real wipe, not a soft delete. Data export is a first-class feature, not a buried setting. Re-ingestion is always possible because the platforms are the source of truth; we are a derived view.
3. **Local-feeling, cloud-backed.** The desktop app is the primary surface. It feels like a local app — fast, responsive, works offline for already-loaded content — but its data lives encrypted in the cloud so the user can access from multiple devices.
4. **One database, one schema.** All ingested events from all sources unify into a single events table with a consistent shape. Cross-platform threading and search depend on this.
5. **Calm with personality.** The visual design is restrained and productivity-oriented by default — generous whitespace, typography-driven, subtle dense information. Personality is permitted at AI moments (insight surfacing, generation completion, successful share) and reserved for delight; trust-critical surfaces (auth, settings, data management, privacy disclosures) stay serious. The earlier "Arc Browser register" framing has been softened: see Section 8's Tier 3 design direction for current references (Linear, Stripe Atlas, Arc as optional flair only).
6. **Ship the V1, then iterate.** Scope creep is the enemy. Every "while we're at it" is a delay to first useful product.
7. **Engine first, platforms second.** The engine — data model, ingestion pipeline, reasoning logic, generation logic, sharing infrastructure — *is* the product. Platforms (Tauri desktop, Android Kotlin, future iOS, future web) are delivery mechanisms for the same engine. Most product logic belongs server-side (Edge Functions, shared schema) or in shared client logic. Platform-specific code is for rendering and platform-native capture (e.g. SMS via Android content provider). New surfaces should be cheap to add precisely because the engine doesn't move when a new client lands.
8. **Intent-first home that surfaces fresh content.** The home surface centers on what the user wants to do or see right now — generated artifacts, on-this-day surfacing, fresh insights, recently-relevant context. Commands are accessed *through* the home, not the home itself. The timeline is one view among several, not the gravity well. Power-user features (Cmd+K, keyboard nav, advanced commands) are present but not foregrounded — discovery happens for users who go looking. This is Apple's playbook (consumer experience leads, depth is discoverable), not Linear's (power-user experience leads, consumer discovers it later).
9. **Generated output is first-class.** Reports, digests, briefings, collages, and other AI-generated artifacts persist as named entities in the user's archive (the `artifact` table — see Section 6). They are editable, shareable, and part of the data model — not ephemeral chat responses that vanish on refresh. Generation is a creation act, and what's created has a row.
10. **Cross-source reasoning over single-source views.** The product's distinctive value is that no single platform knows what Codex knows. Features that work across sources beat features that work within one. "Brief me on Acme" pulling from Gmail + Calendar + Drive + manual notes is the shape of work that Flashback uniquely enables; "search my Gmail" is something Gmail already does.
11. **Live-API-first with export-zip fallback.** The current platform data landscape favors data sovereignty — most platforms restrict ongoing third-party API access while still honoring data export rights (GDPR / CCPA). Codex prioritizes live API access where it exists, is stable, and is reasonably accessible (current: Google services, Spotify; others meeting these criteria as they surface); export-zip uploads serve as the fallback for the rest. For any single platform, both paths may eventually be supported — live for fast first-experience and ongoing sync, export for completeness and resilience to API changes. The build order per platform is decided per milestone (see Section 12.6); we don't pre-build a unified live/export plugin pattern speculatively.

## 3. V1 Scope

V1 is a **three-tier roadmap** picking up from M12 (Android per-event detail). Each tier is a phase, not a single milestone — each contains multiple milestones to be scoped and numbered as they are prioritized. The order of tiers matters: data density precedes feature work because most engine features (Brief me, generation, cross-source insights) only become interesting once the archive is dense; UI evolution comes last because the new surfaces (intent-first home, artifact rendering, generated-output canvases) can't be designed coherently until the features that fill them exist.

### Platforms (delivery mechanisms — see principle 7)

- **Desktop app:** Windows and Linux first (the user does not own a Mac). macOS support follows in V2. Primary surface for ingestion uploads, generation, sharing.
- **Android app:** Full peer surface. Phone authenticates directly with Supabase, derives keys from passphrase, encrypts client-side, ingests phone-native sources (SMS/calls/contacts/on-device photos), browses timeline (M11) with per-event detail (M12), and will gain feature parity with desktop incrementally.
- **iOS app:** V2 — but architecture must not block it. Engine-first means iOS is a delivery mechanism added cheaply when prioritized.
- **Web app:** Future delivery mechanism. Not in V1; engine should not assume "client = native".
- **Public surface (`apps/public/`):** Cloudflare Pages, separate deploy target. Renders shared artifacts to anonymous viewers. Lives behind its own subdomain (TBD: e.g. `share.flashback.app`).

### Tier 1 — Data density (next 4–6 milestones)

Sources to add before feature work resumes. The archive is currently sparse outside Google + Spotify + manual entries; engine features built on a sparse archive are unconvincing. Tier 1 is now ordered to support the consumer artifact path (year-in-review, on-this-day, photo-rich digests) — photos and location land before message-stream completeness because they're what makes generated artifacts feel emotional and visually distinctive, not just informationally dense.

- **M13 ✓** — SMS + Android Calendar via Android content providers (shipped). Phone-as-peer architecture from M10: Android encrypts client-side and inserts directly to Supabase. Calendar uses the Path C decision (filter `account_type='com.google'` so Google calendars come exclusively through desktop OAuth; Android picks up Samsung-native, Outlook/Exchange, work, LOCAL).
- **M14** — Android gallery (photos with file content) + binary-blob encryption pipeline + foreground gallery sync + background sync infrastructure. **Five-phase milestone with gates.** Introduces Supabase Storage as a target (encrypted blobs), per-blob HKDF-derived keys (Section 6 / 12.6), thumbnail-first storage at 1024px max edge, and `WorkManager`-based background sync with Samsung battery-optimization handling. Background sync infrastructure ships here rather than as a separate later milestone — continuous capture is part of "data density."
- **M15** — Generic Google Takeout pipeline as infrastructure **+ YouTube watch history as the first consumer parser**. Handles the export-zip format and routes per-category data to per-category parsers (TypeScript-first per Section 12.6's escalation rule), and ships the YouTube watch history parser as the validating first consumer so the registry shape is proven against real production data before M16 (Maps Timeline) extends it. Unlocks several Tier 1 sources in one infrastructure milestone (M16 Maps Timeline plus future Takeout-backed sources). *Reshape note (rev 46):* the original M15 description ended at "infrastructure" and deferred all consumers to M16/M17; the M15 investigation (rev 46) folded YouTube watch history ingestion into M15 because shipping infrastructure without a real consumer doesn't validate the registry shape, and consequently M17 was reduced to YouTube downstream features (see M17 below).
- **M16** — Maps Timeline via device-side Android export. - **M17** — YouTube downstream features (channel attribution, watch-time aggregates, year-in-review YouTube section). *Reshape note (rev 46):* originally scoped as "YouTube watch history via Takeout", but the M15 investigation (rev 46) folded the ingestion half into M15 so the registry shape validates against a real consumer; M17 reduces to the surface-features half that builds on M15's already-ingested events.
- **(Future Tier 1)** — Additional sources surface as user need is demonstrated rather than pre-scheduled. **WhatsApp** is deferred until the user actually uses WhatsApp (the parser work is non-trivial and the per-chat export model is friction-heavy; not worth scheduling speculatively). **Instagram** is deferred to demonstrated demand (Graph API requires Meta business verification, Basic Display sunset 2024, export-zip with 24–72hr delay; see Section 11).

### Tier 2 — Engine features (after Tier 1)

Features that exploit the now-dense archive. These are why the data density work happens — they're where Flashback's distinctive value shows up. Ordered to put the flagship signature artifact first, then the public-sharing infrastructure that lets it travel, then the AI / generation features that build on both.

- **Year-in-review as flagship signature artifact** — Felton-style annual report (magazine-spread typography, dense data presentation in calm layouts, numbers as visual anchors) crossed with Spotify Wrapped's universal-relatability and shareability. **Probably the largest single feature milestone** — the rendering layer is real design discipline, not template work. Anchors the consumer-positioning thesis: demonstrates breadth of ingestion (every Tier 1 source feeds it), visual distinctiveness, universal relatability ("everyone has a year"), and natural shareability. First public consumer-facing surface; sets the bar for every subsequent generated artifact. The artifact rendering must handle sparse-archive gracefully so an early-onboarded user with one or two connected sources still gets something coherent rather than a broken layout.
- **Public sharing — single entry MVP** — toggle one event or artifact to a public URL with slug-based addressing. - **Public sharing — curated digests** — multiple events grouped, named, published as a single shared artifact.
- **"Brief me"** — single highest-leverage AI feature. "Brief me on Jordan" → structured briefing pulling from every source involving Jordan, with citations. Designed-page output (consistent with the year-in-review aesthetic), not chat-text.
- **Threads** — persistent entity-centric views (people, projects, topics) that survive across sessions and can be opened from any related event.
- **Generation workspace** — one-prompt artifact creation (collage, expense summary, slide-deck compilation, highlight reel).
- **Smart digests on schedule** — weekly/monthly auto-generated summaries, persisted as `artifact` rows.
- **Cross-source insights surfaced ambiently** — "you've mentioned Acme 47 times this month across email and calendar" without the user asking.
- **Audit trail for AI-generated outputs** — every claim in a generated artifact cites the events that led to it; user can audit any output back to source.

Visual and interaction redesign once features have stabilized. ### Cross-cutting throughout all tiers

## 4.5 Android Code Conventions

Apply these conventions to all code under `apps/android/`. They exist so Cursor (and humans) write the Android client against an intentional set of choices, not whatever each tool reflexively suggests. The directive cliff-notes version lives at `apps/android/.cursorrules`; this section is the canonical, fuller form.

### Language and idioms
- **Kotlin only.** No Java unless required by an Android API that has no Kotlin equivalent.
- **Prefer Kotlin idioms:** data classes, sealed classes, `when` expressions, scope functions, extension functions where they add clarity.
- **Coroutines for all async work.** No callbacks, no RxJava, no raw threads.
- **`Flow` for streams of values, `suspend` functions for one-shot async operations.**
- **Null safety:** avoid `!!` (the not-null assertion). If a value might be null, handle it explicitly with `?:`, `?.let { }`, or an early return.

### UI
- **Jetpack Compose only.** No XML layouts, no `View`s.
- **Material 3 components.** No Material 2.
- **Stateless composables where possible** — pass state in, emit events out via lambdas.
- Use `rememberSaveable` for state that should survive config changes; `remember` for transient UI state.
- Composables follow `PascalCase` and are nouns or noun phrases (`HomeScreen`, `EventCard`, `AuthForm`).
- Long composables (>100 lines) should be decomposed.

### Architecture
- **MVVM with one ViewModel per screen.**
- ViewModels expose `StateFlow<UiState>` for state; accept events via methods.
- `UiState` is a single sealed class or data class per screen (`Loading`, `Success`, `Error`, etc.).
- **Repository pattern for data access** — composables don't talk directly to Supabase or storage.
- **Single Activity (`MainActivity`)**, all navigation via Compose Navigation.

### Dependency injection
- **Manual constructor injection while the graph is small.** M10 ships ~7 classes (Supabase client + repositories + session store + ViewModels) with `lazy` properties on `CodexApplication` and `viewModels { Factory(...) }` per ViewModel. Migrate to Hilt when the graph crosses ~15–20 classes or scoped lifecycles tied to navigation graphs are needed (see Section 12.3). Don't bundle the migration with feature work — separate refactor milestone.

### Crypto
- **All crypto operations go through the central crypto module** at `apps/android/app/src/main/java/com/codex/companion/crypto/`.
- **Never expose raw key material outside this module.**
- **All crypto constants live in `CryptoConstants.kt`** — single source of truth that mirrors the desktop's Rust constants exactly.
- **Cross-platform test vectors at `packages/shared/test-vectors/` are the regression test for parity** — any change to crypto must update the vectors and pass tests on both platforms.
- **Library choices (rev 20):** Bouncy Castle's `Argon2BytesGenerator` for Argon2id key derivation; **JDK `javax.crypto.Cipher`** directly for AES-256-GCM (chosen over Tink's `AesGcmJce` because Tink's encrypt API has no public hook for a fixed nonce, which the parity test pipeline needs); **Tink** (`com.google.crypto.tink.subtle.Hkdf`) for HKDF-SHA256 (the JDK has no native HKDF); JDK `javax.crypto.Mac` for HMAC-SHA256. The deviation from "Tink for AEAD" is intentional and documented inline in `PayloadCrypto.kt`; the underlying primitive (AES-256-GCM with 12-byte nonce, 128-bit tag, no AAD, `nonce ‖ ciphertext ‖ tag` format) is identical.

### Permissions
- **Request runtime permissions only when the feature needs them**, not all at app start.
- **Always handle permission-denied gracefully** — show explanatory UI; link to system settings if the user denied permanently.
- **Foreground services for long-running operations require ongoing notifications** — don't add foreground services until there's a real reason.

### Testing
- **Unit tests for crypto, repositories, and ViewModels.**
- **JUnit 5 (`jupiter-api`) for new tests.**
- **MockK for mocking, not Mockito.**
- **Coroutine tests with `kotlinx-coroutines-test`.**
- Composable previews are nice-to-have but not required for every component.

### Dependencies
- **Use the Gradle version catalog (`gradle/libs.versions.toml`)** — no inline version strings in `build.gradle.kts` files.
- **Pin all versions explicitly.**
- **Don't add a new dependency without good reason** — prefer the standard library.

### File and package organization
- **Package by feature, not by type:**
  - `auth/` contains `AuthScreen`, `AuthViewModel`, `AuthRepository`, `SupabaseAuth` — everything auth-related together.
  - **NOT** `screens/AuthScreen` + `viewmodels/AuthViewModel` + `repositories/AuthRepository`.
- Keep packages shallow; deeply nested packages are usually a smell.

### Build and lint
- **Kotlin formatting via ktlint or detekt** — pick one and configure CI.
- **Lint runs as part of the build**; warnings should be addressed, not suppressed.
- **ProGuard/R8 enabled for release builds**; debug builds skip shrinking.

### What NOT to do
- **No copying TypeScript code idioms** — the React patterns from desktop don't translate to Compose 1:1.
- **No reaching for popular libraries Cursor might suggest reflexively:** avoid Retrofit (we have the Supabase Kotlin SDK), avoid Room (no local DB needs yet), avoid Glide/Picasso (`AsyncImage` from Coil is the modern choice if image loading becomes necessary).
- **No `SharedPreferences` for sensitive data** — no session tokens, derived keys, or user content. Use `EncryptedSharedPreferences` (`androidx.security:security-crypto`) backed by `MasterKey.Builder` (modern API) for sensitive KV storage; use Android Keystore directly for raw key material that needs hardware-backed protection.
- **No telemetry or crash reporting libraries** until we consciously decide to add them.
- **No analytics SDKs.**

### Events (the heart of the schema)

```sql
create table public.event (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  source_connection_id uuid references public.source_connection(id) on delete cascade,
  source text not null,                    -- denormalized from source_connection for query speed

  -- WHEN
  occurred_at timestamptz not null,        -- when the event happened in real life
  occurred_date date generated always as ((occurred_at at time zone 'UTC')::date) stored,
  ingested_at timestamptz not null default now(),

  -- WHAT
  event_type text not null,                -- 'message', 'photo', 'post', 'comment', 'like', 'listen', 'watch', 'location', 'call', 'sms'

  -- ENCRYPTED CONTENT (server cannot read these)
  encrypted_payload text not null,         -- JSON blob encrypted with user's data key, contains: text, participants, metadata
  encrypted_search_terms text,             -- Encrypted FTS-friendly tokens (see search section)

  -- UNENCRYPTED METADATA (used for filtering/sorting; chosen carefully to leak minimal info)
  participant_hash_set text[] default '{}',-- Deterministic hashes of participant identifiers (allows "find by person" without revealing who)
  has_attachments boolean not null default false,
  byte_size int not null default 0,        -- For storage analytics

  -- EMBEDDING (encrypted? See note below)
  embedding vector(1024),

  -- DEDUPE
  source_event_id text,                    -- ID from the source platform, used for dedupe on re-ingestion
  content_hash text,                       -- Hash of normalized content for cross-source dedupe

  unique (user_id, source, source_event_id)
);

create index event_user_occurred_idx on public.event (user_id, occurred_at desc);
create index event_user_date_idx on public.event (user_id, occurred_date desc);
create index event_user_type_idx on public.event (user_id, event_type);
create index event_user_participant_idx on public.event using gin (participant_hash_set);
create index event_embedding_idx on public.event using hnsw (embedding vector_cosine_ops);
```

**Critical notes on encryption + search:**

The fundamental tension: full encryption means the server can't search the content. We resolve this with two techniques:

1. **Encrypted search tokens.** Client tokenizes content (lowercased words, normalized), then for each token, computes `HMAC(token, user_search_key)` where `user_search_key` is derived from the user's master key. These deterministic hashes are stored in `encrypted_search_terms` (an array of HMACs). At query time, the client HMACs the search query terms and looks for matches. This gives us encrypted full-text search. Limitations: no fuzzy matching server-side, no stemming server-side. Client can do those before HMACing for known-good cases.
2. **Embeddings.** Generated server-side via Voyage AI (V1) or client-side using a small model (V2). Embeddings are *not* encrypted in V1 — this is a deliberate compromise documented in `ENCRYPTION.md`. The embedding represents the event's meaning numerically; while not as sensitive as plaintext, it does leak semantic information. A V1 acceptable tradeoff with explicit user disclosure. V2 should move to client-side encrypted embeddings or a privacy-preserving similarity scheme.

```sql

  user_id uuid not null references auth.users(id) on delete cascade,
  kind text not null,               -- 'briefing', 'digest', 'report', 'generation', 'thread'
  title text not null,
  -- ENCRYPTED CONTENT (server cannot read these)
  encrypted_payload text not null,  -- JSON of the artifact's structured content, encrypted with user's Kd
  encrypted_prompt text,            -- The user's prompt that generated this, encrypted; nullable.
                                    -- Stored for retroactive understanding of how an artifact was created.
  -- PROVENANCE
  event_ids uuid[] default '{}',    -- Events that contributed to generation (citations)
  -- TIMING
  generated_at timestamptz not null default now(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index artifact_user_generated_idx on public.artifact (user_id, generated_at desc);
create index artifact_user_kind_idx on public.artifact (user_id, kind);
```

RLS: users read/write their own rows only. **No anonymous access** — public sharing of an artifact happens by writing a derived row to `shared_artifact`, never by exposing `artifact` itself. This is intentional: drafts of generated content (e.g. multiple briefing variants the user discards) must remain private even when the user later chooses to share one.

### Public shared artifacts (Tier 2+)

Where public sharing lives. **Purposefully separate from the encrypted `event` and `artifact` tables — content here is cleartext** because anonymous viewers must be able to read it without a key. The user explicitly opts each row in (Section 7's "Public sharing and the encryption boundary").

```sql
create table public.shared_artifact (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  -- ADDRESSING
  slug text not null unique check (
    length(slug) between 1 and 100
    and slug ~ '^[a-z0-9-]+$'
  ),                                -- Short URL-safe identifier, e.g. 'q1-acme-update'.
                                    -- Lowercase alphanumeric + hyphens only; 1-100 chars.
                                    -- Prevents URL-encoding edge cases and abuse via overlong slugs.
  -- CONTENT (CLEARTEXT — anonymous viewers must be able to read this)
  title text not null,
  description text,
  kind text not null,               -- 'single_event', 'digest', 'report', 'thread_snapshot'
  payload jsonb not null,           -- Cleartext content of the shared artifact:
                                    --   single_event   = the rendered event
                                    --   digest/report  = the structured artifact
                                    --   thread_snapshot = grouped event renderings
  event_ids uuid[] default '{}',    -- References to original events (provenance + backlinks)
  -- LIFECYCLE
  published_at timestamptz,         -- null = draft, set when published
  revoked_at timestamptz,           -- set when user unpublishes; takes precedence over published_at
  -- ANALYTICS (cleartext is fine — this is metadata about a public URL)
  view_count int not null default 0,
  last_viewed_at timestamptz,
  -- TIMING
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create unique index shared_artifact_slug_idx on public.shared_artifact (slug);
create index shared_artifact_user_idx on public.shared_artifact (user_id, created_at desc);
create index shared_artifact_public_visible_idx on public.shared_artifact (slug)
  where published_at is not null and revoked_at is null;
```

RLS for `shared_artifact` is unique in this schema:
- The owner (`auth.uid() = user_id`) can read, write, update, delete — same as every other table.
- The **anonymous role** can read rows where `published_at is not null and revoked_at is null` — and can read *only those* columns needed for rendering (cleartext columns are by definition fine to expose).

Note on the `event` table: events can have zero or many references in `shared_artifact` and `artifact` via the `event_ids` arrays. **No schema change to `event` is required** — the relationship is one-way (artifacts/shares point at events, not the other way around) and rebuilds easily by querying. If reverse-lookup (e.g. "is this event currently shared?") becomes hot, add a materialized view or a join table later; not needed for V1.

### Encrypted blob storage (Tier 1 — M14)

Photos, MMS attachments, voice memos, and other binary content live in **Supabase Storage** with client-side encryption. This pattern is *distinct from* the `encrypted_payload` column on event rows (which holds JSON-shaped content) for two reasons:

1. **Size.** Event payloads are JSON-shaped and small (~10KB max); blobs are binary and large (50KB–50MB; for V1's thumbnail-first ingestion, ~150KB–5MB).
2. **API shape and wire format.** Payloads use single-shot AES-GCM and embed the nonce in the ciphertext (`nonce ‖ ciphertext ‖ tag`). Blobs use a **chunked API surface backed by single-shot AES-GCM encryption in V1** — the chunk_size parameter affects API buffering only, **not output bytes**. Wire format is single-nonce + trailing-tag (the 16-byte authentication tag at the end of the ciphertext), with the nonce stored separately on `blob_ref.encryption_nonce` rather than prepended. The wire format is **forward-compatible with future true-streaming implementations** when blob sizes warrant it (Section 12.3 tracks the V2 upgrade path). For V1's thumbnail-first ingestion (~150KB–5MB blobs), full-plaintext-in-memory during encrypt/decrypt is acceptable on both Android (modern phones have multi-GB heap) and desktop. The recommended chunk size for the chunked API surface is ~64KB (used in test vectors and as the canonical production default), but any chunk_size produces identical output bytes.

**Schema additions:**

A new `blob_ref` column on the `event` table (jsonb, nullable) — when an event has associated binary content, the column points at the encrypted blob in Storage and carries the metadata needed to render and decrypt it without a round trip:

```jsonc
{
  "storage_path": "...",         // path in Supabase Storage (per-user prefix, blob UUID filename)
  "encryption_nonce": "...",     // base64 nonce for AES-GCM
  "encrypted_size_bytes": 0,     // for transfer cost estimation / progress UI
  "decrypted_size_bytes": 0,     // for layout pre-allocation hints
  "content_type": "image/jpeg",  // 'image/jpeg', 'image/heic', 'image/png', etc.
  "width": 0,                    // for layout pre-allocation (avoids reflow on decode)
  "height": 0                    // for layout pre-allocation
}
```

**Per-blob encryption key derived via HKDF from the user's data key Kd:**

```text
blob_key = HKDF-SHA256(
  ikm    = Kd,
  salt   = (none),
  info   = 'codex.blob' || blob_uuid,
  length = 32
)
```

**Reasoning:** AES-GCM requires unique nonce-key pairs — two blobs sharing a nonce under the same key would compromise both. Per-blob keys eliminate that entire failure class structurally rather than by careful nonce bookkeeping. HKDF derivation is essentially free (already in our crypto facades from M2 — the search key Ks is derived the same way, see Section 7). The blob UUID becomes part of the HKDF `info` parameter, making the derivation deterministic from the row itself; no per-blob metadata needs to be stored to recover the key on decrypt.

**UUID string form invariant:** the `derive_blob_key` facade performs **no normalization** on the input string — uppercase vs. lowercase, hyphenated vs. unhyphenated, or padded representations all produce *different* derived keys. Production code MUST use the **canonical lowercase hyphenated UUID v4 form** (36 chars: 8-4-4-4-12 hex digits, lowercase, hyphens at positions 8/13/18/23 — i.e., what `UUID.toString()` returns in Java/Kotlin and `Uuid::to_string()` returns in Rust with the standard formatter) when both encrypting and decrypting a given blob. The cross-platform test vectors (`packages/shared/test-vectors/crypto-vectors.json`) pin lowercase hyphenated as the production form; the lone `"x"` test vector exists to prove the derivation is purely byte-driven and to catch any accidental UUID-format normalization on either platform.

**Empty UUID policy:** `derive_blob_key("")` is **valid** in V1 — the HKDF info reduces to just the 10-byte `b"codex.blob"` prefix. This is a permissive design choice (HKDF is byte-driven; rejecting empty strings would require special-case validation that doesn't add cryptographic security), pinned by an inline `blob_aead` unit test rather than a JSON vector. Production code paths should not pass empty UUIDs in normal operation — every blob has a real UUID — but the function does not enforce this.

**Storage RLS:** only the owning user can read their own blobs. The Supabase Storage bucket policy mirrors the `event` table's `user_id = auth.uid()` rule, scoped on the per-user path prefix. Public sharing of blob content is a **future consideration** (likely involves re-encryption with a share-specific key, or copying decrypted bytes to a separate public bucket); not in scope for M14. The Tier 2 public-sharing milestones currently address text-shaped artifacts only.

**Storage path convention:** blobs are stored at paths of the form `<user_uuid>/<blob_uuid>` (per-user prefix as the first path segment, blob UUID as the filename, no extension since the on-wire bytes are encrypted ciphertext rather than a recognizable image format). RLS policies on `storage.objects` filter by `(storage.foldername(name))[1] = auth.uid()::text`, ensuring users can only access objects under their own prefix. This convention is established by the M14 phase 2 server-setup migration (`supabase/migrations/0006_blob_storage.sql`) and verified at policy-application time; client code on both Android and desktop MUST construct upload paths using this exact shape or the upload will be rejected by RLS.

**Relationship to Section 7's encryption hierarchy:** the per-blob key Kb is a sibling of the existing search key Ks — both are HKDF-derived from Kd. See Section 7's key hierarchy for Kb's place in the broader key derivation chain.

### Row Level Security

Every table above has RLS enabled with a policy: `user_id = auth.uid()`. There are no admin-readable rows. There is no exception to this rule.

```sql
alter table public.user_profile enable row level security;
alter table public.source_connection enable row level security;
alter table public.event enable row level security;
alter table public.attachment enable row level security;
alter table public.ingestion_job enable row level security;
alter table public.artifact enable row level security;
alter table public.shared_artifact enable row level security;

create policy "user_profile_owner" on public.user_profile
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());

create policy "source_connection_owner" on public.source_connection
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());

create policy "event_owner" on public.event
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());

create policy "attachment_owner" on public.attachment
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());

create policy "ingestion_job_owner" on public.ingestion_job
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());

create policy "artifact_owner" on public.artifact
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());

-- Owner can do anything with their own shared_artifact rows.

  for all using (user_id = auth.uid()) with check (user_id = auth.uid());
```

## 7. Encryption Architecture

Detailed in `docs/ENCRYPTION.md` but summarized here:

### Key hierarchy
1. **Passphrase** — what the user types to log in. Sent to Supabase Auth for authentication. Also used as input to KDF on client.
2. **Auth password hash** — held by Supabase Auth. Standard bcrypt. Different from KDF output (different salt + algorithm).
3. **Master key (Km)** — derived client-side from passphrase using Argon2id. Never sent to server. Lives in memory and OS keychain only.
4. **Data key (Kd)** — random 256-bit AES key generated at signup. Encrypts all user content. Stored on server as `wrapped_data_key` = `AES-GCM(Km, Kd)`. Server cannot decrypt because server doesn't have Km.
5. **Search key (Ks)** — derived from Kd using HKDF. Used for HMACing search tokens.
6. **Per-blob key (Kb, M14+)** — derived from Kd using HKDF-SHA256 with `info = 'codex.blob' || blob_uuid`, length 32 bytes, no salt. One key per blob, derived deterministically from the blob's UUID stored on the owning `event` row. Used for streaming AES-GCM encryption of binary blobs (photos, MMS attachments, voice memos, etc.) stored in Supabase Storage. Per-blob derivation eliminates the AES-GCM nonce-reuse failure class structurally — two blobs cannot share a nonce under the same key because they don't share a key. Lands when M14 phase 1 (binary-blob pipeline foundation) ships; documented here so the hierarchy is complete. See Section 6's "Encrypted blob storage" for the schema and Section 12.6 for the architectural rationale.

### What gets encrypted (client-side, with Kd)
- All event payload content
- All attachment file contents
- All attachment metadata
- OAuth tokens (so server can't impersonate user to Google etc.)

### What is NOT encrypted (server can see)
- User email and auth identifiers (Supabase Auth holds these)
- Event timestamps, source platform, event type, attachment count, byte size
- Event embeddings (V1 compromise; V2 priority to fix)
- Participant identifier hashes (deterministic HMACs — server can correlate but not identify)
- Ingestion job status

### Login flow
1. User enters email + passphrase
2. Supabase Auth validates credentials
3. Client fetches `user_profile` (gets `wrapped_data_key`, KDF params)
4. Client derives Km from passphrase + KDF params
5. Client unwraps Kd by `AES-GCM-decrypt(Km, wrapped_data_key)`
6. Kd is held in memory and (optionally) OS keychain for the session

**Fail-closed rule:** If `user_profile` does not exist for an authenticated user, signin must fail closed with an error and sign the user out of the Supabase session — rather than auto-creating keys. Auto-creation of encryption keys is only valid in the signup flow, where we know the account is fresh and has no existing encrypted data. Silently generating new keys on signin would make any existing encrypted archive permanently unreadable.

### Password change (current passphrase known)
1. Derive new Km from new passphrase
2. Re-wrap existing Kd with new Km
3. Update `wrapped_data_key` and KDF params on server
4. Update auth password via Supabase Auth

### Password reset (current passphrase NOT known)
1. User clicks "forgot password," receives email link
2. Client warns: *"Resetting your passphrase will create a new encrypted archive. Existing data on our servers will be deleted (it can no longer be decrypted). You'll need to reconnect your accounts and re-import any uploaded files."*
3. On confirmation, server marks all events/attachments/connections for deletion (7-day grace), creates new empty user_profile with new wrapped_data_key derived from new passphrase
4. User starts fresh

### Public sharing and the encryption boundary (Tier 2+)

Public sharing is a **deliberate exit from the encryption envelope**. When a user toggles content public, a *cleartext copy* is written to the `shared_artifact` table (Section 6). This is not a weakening of the privacy guarantee — the original encrypted `event` (or `artifact`) row remains encrypted; the user has explicitly opted to publish a derivative.

**Two guarantees coexist:**

1. *Only you can read your private archive.* Unchanged from V1's encryption posture. The `event`, `attachment`, and `artifact` tables are encrypted client-side with the user's data key (Kd). Server cannot decrypt any of them.
2. *Shared content is exactly what you chose to share, no more.* Shared rows in `shared_artifact` are cleartext by design (anonymous viewers can't be given a key), and they are written only by the owner client at explicit user request. Nothing in the encrypted archive ever reaches `shared_artifact` automatically.

**Why `artifact` is encrypted even though the user might share some artifacts later:** generated content is sensitive. The user may generate ten draft briefings of "everything I've exchanged with Jordan about the Acme deal" before publishing one. The other nine drafts must remain private. Encryption-by-default for `artifact` (rather than "encrypted only if not yet shared") is the simpler and stronger guarantee.

**Lifecycle of a publish:**

1. User selects an event, digest, or artifact and clicks "Share publicly".
2. Client decrypts the source content with Kd (already in memory; the user is signed in).
3. Client renders the content into the cleartext payload shape that `shared_artifact.payload` expects.
4. Client writes a `shared_artifact` row with `published_at = now()`, `revoked_at = null`. The `event_ids[]` array preserves provenance back to the originals.
5. The public URL (`/<slug>`) is now resolvable on the public surface (`apps/public/`) via the anonymous-read RLS policy.

**Lifecycle of an unpublish:**

1. User clicks "Revoke" on a shared artifact.
2. Client updates the row, setting `revoked_at = now()`.
3. The anon-read RLS policy stops returning the row immediately. Cached HTML at the Cloudflare edge may persist briefly; the public surface respects short cache TTLs for shared content for this reason.
4. The row is retained (not deleted) so the user can republish later or audit what was once public.

Hard-delete (truly remove a previously-shared artifact, not just revoke) is deferred to a follow-up. **V1 sharing is revoke-only**; users who need to truly delete a previously-shared artifact in V1 should contact the founder out-of-band. The Tier 2 follow-up to ship hard-delete is tracked in Section 12.3 — required before any non-founder users are onboarded to sharing, since revoke leaves the cleartext payload in our database and that violates user expectation that "I can take it down completely."

**What the server can see for shared content:** everything in `shared_artifact.payload`. This is a deliberate trade. The user is told this clearly at the publish moment ("This will be visible on the public internet at this URL. Codex's servers will be able to read it. Anyone with the link will be able to read it.") and they confirm before the row is written.

## 9. Key User Flows

### First-run onboarding (desktop app)
1. Splash with brand mark + tagline
2. Sign up: email + passphrase + (optional) Google sign-in or phone OTP
3. Passphrase strength guidance (zxcvbn or similar)
4. Encryption setup: client derives Km, generates Kd, wraps it, sends to server. Brief animation explaining "your archive is being secured" — should feel substantive but be quick.
5. "Connect your first source" — present the six V1 sources with brief descriptions and "Connect" buttons. User can skip and add later.
6. After first connection: ingestion runs in background, user sees a progress view with playful animations (data flowing in, day cards arriving)
7. Once first batch ingested: user lands in the timeline, with their first day populated. **This is the wow moment.** Design effort goes here.

### Connecting a source via OAuth (e.g., Google)
1. User clicks "Connect Google" in Sources
2. Tauri opens system browser to Supabase Edge Function `oauth-google/start` which redirects to Google
3. Google authorizes, redirects back to Supabase Edge Function `oauth-google/callback`
4. Edge Function receives token, returns it to desktop app via deep link or LAN callback
5. Desktop app encrypts the token with Kd, sends to server as `encrypted_credentials` on a new `source_connection` row
6. Server schedules first ingestion job

### Uploading an export (e.g., Instagram)
1. User clicks "Upload Instagram export" in Sources
2. UI explains how to request export from Instagram (link to instructions)
3. User drops zip file onto upload zone
4. Desktop app parses the zip locally (using `packages/parsers/instagram`), produces structured events
5. For each event: encrypt payload with Kd, generate search tokens, hash participants, send to server
6. Attachments uploaded to Supabase Storage (encrypted)
7. After all events ingested, trigger embedding job

### Daily timeline view
1. Default view on app open
2. Reverse chronological, day-grouped
3. Each day = a section header (date, weather if known, # of events) + a vertical layout of events
4. Events render differently by type:
   - Messages: chat-bubble style with participant avatar
   - Photos: full-width cards, single or grid
   - Posts: source-styled cards (e.g., Instagram-tinted)
   - Listens: compact row with album art
   - Locations: small map preview
5. Infinite scroll back through history
6. Mini-map / scrubber on right edge for fast year-jumping

### Search
1. Persistent search bar (Cmd/Ctrl+K to focus from anywhere)
2. As user types: instant results (full-text via encrypted search terms)
3. After short pause (300ms idle): also runs semantic search, merges results with FTS
4. Results group by relevance, with date + source badges
5. Filters: date range, source, event type, person — all applied via metadata fields, not content
6. Empty state and zero-results state both have personality

### Per-person threads
1. From any event, click a participant → enter their thread view
2. Shows all events involving that participant, across all sources, chronologically
3. Includes: messages, mentions in posts, photos featuring them (if tagged), shared events
4. Person identity is resolved client-side (client groups participant identifiers that look like the same person)

### Ask
1. Persistent "Ask" affordance (sparkle icon, cmd/ctrl+; keyboard shortcut)
2. User types question
3. **First time per session:** a non-blocking modal explains: *"To answer, we'll send relevant events from your archive to Anthropic to generate a response. We send only what's needed to answer your question. Anthropic does not train on this data. Your archive itself stays encrypted on our servers."* with a "Got it" / "Cancel" option.
4. Client embeds the question (inheriting prior conversation context for follow-ups) → retrieves top-30 events via semantic search → decrypts → sends conversation history + events to Edge Function `ask` → Edge Function calls Claude → streams answer
5. Answer streams in, with cited events shown per turn
6. Each cited event is clickable to jump to its detail in the timeline
7. Multi-turn: user can ask follow-up questions in the same conversation. Retrieval inherits context from the prior turn. Claude receives the last 5 turns of conversation history. A visual "Earlier in this conversation..." marker separates turns outside Claude's context window from those inside it.
8. Hard cap of 10 turns per conversation. At cap, input disables and user is prompted to start a new conversation.
9. Conversations are memory-only (Zustand, transient) — discarded on app restart or sign-out. Persistent conversations deferred to V1.1.

### Account deletion
1. Settings → Danger Zone → Delete Account
2. Modal with clear warning, user must type their email address to confirm (paste disabled)
3. On confirm: client calls `delete-account` Edge Function with action `schedule`, which sets `user_profile.deletion_scheduled_at = now()`
4. User is signed out immediately
5. During 7-day grace: if user signs back in, a banner at top of app shows scheduled deletion date with a "Cancel deletion" button. Cancel calls `delete-account` with action `cancel`, which clears `deletion_scheduled_at`.
6. After 7 days: `purge-deleted-accounts` Edge Function (triggered daily at 3am UTC via `pg_cron`) calls `auth.admin.deleteUser()` for expired accounts, cascading to all dependent tables (`user_profile`, `event`, `attachment`, `source_connection`, `ingestion_job`, `artifact`, `shared_artifact`)

### Public sharing (Tier 2+ — sketch)

Full spec lands at the milestone. Skeleton:

1. From an event detail, an artifact, or a multi-select of events, user clicks "Share publicly".
2. Client opens a "Preview public rendering" modal showing exactly what the public surface will display, including any redactions / formatting transformations.
3. Modal includes: editable `title`, `description`, `slug` (auto-suggested, user-editable, uniqueness-checked); a clear privacy disclosure ("Anyone with the link will be able to read this. Codex's servers will be able to read it.") with an explicit **Confirm publish** button.
4. On confirm: client decrypts the source content and writes a `shared_artifact` row with `published_at = now()`.
5. User gets a shareable URL of the form `https://share.flashback.app/<slug>` (subdomain TBD), copy-to-clipboard affordance, and the row appears in a "Published" list under Sources or a future Public surface in the desktop app.
6. **Revoke**: from the same list, "Unpublish" sets `revoked_at = now()`. URL stops resolving immediately on the public surface (subject to short edge cache TTL).
7. **Edit**: the user can re-encrypt and re-publish a new version — for V1 the simplest model is "revoke + republish with same slug" rather than versioned history.

**Copy specificity at the publish-confirmation step:** the disclosure modal must match the wording specificity of Section 7's encryption-boundary section — *"Codex's servers will be able to read it. Anyone with the link will be able to read it."* Do **not** soften this language for friendliness in the implementation. The explicit acknowledgment of server-side readability is the key trust moment; the user is opting out of the encryption guarantee they signed up for, and the UI must be honest about it. This is a non-negotiable copy requirement that the milestone spec inherits, not something to be A/B tested or "made warmer."

### "Brief me" (Tier 2 — sketch)

The single highest-leverage AI feature. Full spec at milestone time.

1. User invokes "Brief me on [entity]" — either by typing in the command surface or selecting a person/project/topic from a thread view.
2. System resolves the entity (person, project, topic) to a set of related events via participant-hash matches, semantic similarity, and any existing thread-membership data.
3. System retrieves all relevant events, decrypts client-side, and sends a structured generation prompt to the `ask`-style Edge Function (or a new `generate` function — TBD at milestone).
4. Claude returns a structured briefing: sentiment, recent activity timeline, open threads / questions, key participants, citations to source events.
5. Client renders the briefing, persists it as an `artifact` row (kind `briefing`), and offers Edit / Save / Share actions.
6. The briefing's `event_ids` array preserves provenance — every claim can be audited back to its source event.

### Generation workspace (Tier 2 — sketch)

One-prompt artifact creation. Full spec at milestone time.

1. User opens the Generation workspace and types a prompt: "Make me a Q1 client digest for Acme" or "Compile a highlight reel of YouTube videos I watched while commuting last month".
2. System retrieves relevant events (semantic + structured filters from the prompt — date range, source filter, participant), decrypts client-side, and dispatches to the appropriate generation pipeline (text summary → Claude; collage → photo-only retrieval + layout engine; export bundle → zip of decrypted content; etc.).
3. Output is rendered in the workspace, persists as an `artifact` row (kind appropriate to the output: `generation`, `report`, `digest`).
4. User can iterate (refine prompt, re-generate), edit the output directly, save, or share.
5. Each generation captures the user's prompt encrypted in `artifact.encrypted_prompt` for retroactive understanding ("how did I make this?") and re-runs.

## 11. Ingestion Architecture

### OAuth-based sources (Google, Spotify)
- Initial sync: fetch last 5 years of data, ingest in chronological batches of ~500 events
- Incremental sync: scheduled every 6 hours via Supabase cron, fetches only new events since `last_sync_at`
- Token refresh handled by Edge Function

### Export-based sources (Instagram, WhatsApp)
- User uploads zip via desktop app
- Parsing happens **client-side** in Tauri (TypeScript parsers in `packages/parsers/` by default, with per-category Rust escalation when memory or perf demands it — see Section 12.6's "Takeout parsers are TypeScript-first" entry for the escalation rule)
- This is critical: export files contain plaintext that should never reach our server unencrypted
- Parsed events are encrypted client-side, then sent to server via standard event-creation API

### Android device sources (SMS/MMS/calls/contacts/photos)
- The Android app is a peer surface to desktop, not an ingestion-only client. It authenticates directly with Supabase using the same email + passphrase as desktop.
- The Android app derives the master key from the user's passphrase using identical Argon2id parameters as desktop, unwraps the data key from `user_profile.wrapped_data_key`, and runs the full encryption pipeline locally in Kotlin (using Tink for AEAD/HKDF, Bouncy Castle for Argon2id).
- For each Android-native source (SMS, MMS, call log, contacts, on-device photos): the app reads via the appropriate content provider, encrypts each event client-side with the user's data key, and inserts directly into the `event` table via the Supabase client. RLS enforces user scoping.
- Cross-platform test vectors at `packages/shared/test-vectors/` ensure that Kotlin's crypto output matches Rust's bit-for-bit. Any divergence breaks decryption and is treated as a milestone-blocking bug.
- No LAN pairing, no desktop relay, no mDNS discovery. The earlier V1 design (Android relays through desktop) was abandoned in rev 19 because the LAN-pairing UX was a blocker for real users. The simpler architecture (phone is a peer of desktop) is now the V1 design.

### Embedding generation
- After events are ingested, an `embed-events` job runs (server-side, via Edge Function on cron)
- For events missing embeddings: server generates embedding via Voyage AI (`voyage-3`, 1024 dim)
- V1: server-side embedding generation, with explicit user consent disclosed in onboarding privacy summary
- V2: move to client-side embeddings (smaller models, ONNX runtime)

### Per-source landscape (Tier 1 and Tier 2+)

Per principle 11 (Section 2), the question "live API or export?" is answered per-platform based on what each platform actually exposes. The mini-specs below capture the *current landscape* — the access paths that exist and the build path Codex plans to take. They are not commitments to ship both paths in V1 for any platform; they document the menu so that whoever scopes a milestone has the trade-offs in front of them.

Sources we have actively discussed but not yet shipped, grouped by tier and ordered within each tier by current priority. Sources we have not discussed (LinkedIn, Apple, Strava, etc.) are deliberately omitted until they come up — when they do, they get a mini-spec here.

#### Tier 1 — data density (next ingestion milestones)

- **SMS**
  - **Live API:** none. SMS is not exposed to third parties via Google or Apple APIs; what export tooling exists is third-party-app-generated CSV / XML, not platform-native.
  - **Content provider (Android):** `content://sms/inbox`, `content://sms/sent`, `content://mms/...` — the only viable read path on Android. iOS Messages is locked; iMessage ingestion requires Mac and is out of V1 (Section 3).
  - **First build:** Android content provider only, per the phone-as-peer architecture (M10). Encrypted client-side in Kotlin, inserted directly to Supabase via the Android SDK. **Shipped in M13** as the first Tier 1 milestone after M12; message streams are high signal density per row, and Android-native capture was the only viable read path.

- **Android Calendar**
  - **Live API:** Google Calendar API (already shipped via desktop OAuth in M4) covers Google calendars. No live API exists for Samsung-native, Outlook/Exchange, work, or LOCAL calendars.
  - **Content provider (Android):** `CalendarContract.Events` exposes every calendar registered on the device, including Google calendars (synced through Android's account framework) and every non-Google calendar.
  - **First build:** Android content provider, **filtering out `account_type = 'com.google'` at the read layer** (the M13 "Path C" decision). Google calendars come through desktop OAuth exclusively; the Android content-provider path picks up Samsung-native, Outlook/Exchange, work calendars, LOCAL, and any other non-Google `account_type` values. Path A (transform-based dedup against the desktop OAuth feed) was rejected because production currently has only one `google_calendar` event — any dedup heuristic would be designed against an N=1 sample. Known accepted constraint: an Android-only user with no desktop OAuth gets no Google Calendar coverage in V1; not user-surfaced, documented here for when a real user with that profile shows up.

- **WhatsApp** *(future Tier 1 — deferred until the user actually uses WhatsApp)*
  - **Live API:** WhatsApp Business API exists but is restricted to verified business use cases (Meta Business account + app review). No personal-use live access path.
  - **Export:** per-chat "Export chat" feature emits a zip with a `.txt` transcript and media files. User-initiated, per-chat (no bulk-archive export — each conversation is exported separately).
  - **First build:** export-zip only. Parsed client-side in Tauri Rust (`packages/parsers/whatsapp/`), encrypted, inserted via the standard event-creation API. Live API is not viable for personal use. **Deferral note (rev 30):** the per-chat export model is friction-heavy and the parser work is non-trivial; not worth scheduling speculatively. Promotes back into the active Tier 1 sequence when the user starts using WhatsApp themselves or when a real user requests it.

- **Generic Google Takeout pipeline** (infrastructure milestone)
  - Not a single source — it's the export-zip handler that lands as infrastructure and unlocks several sources in one milestone (Maps Timeline, YouTube watch history, Search history, Drive metadata, and other Takeout-backed sources). Routes per-category data to per-category parsers.

- **Maps Timeline**
  - **Live API:** none. Google deprecated the Location History API; Maps Timeline data is no longer available via any live endpoint.
  - **Cloud Takeout (defunct since December 2024):** historically Google Takeout's "Location History (Timeline)" category emitted JSON, KML, or KMZ files (`Records.json` for raw GPS samples — a flat array under `{ "locations": [LocationRecord, ...] }` with hundreds to thousands of entries per day; `Semantic Location History/<year>/<year>_<MONTH>.json` for per-month visit/activity bundles under `{ "timelineObjects": [PlaceVisit | ActivitySegment, ...] }`, the closest legacy analogue to the modern `semanticSegments` shape). Google's December 2024 cloud-side migration moved Timeline data on-device only — for migrated accounts, requesting Location History via Takeout now returns only an `Encrypted Backups.txt` placeholder with no usable JSON; for non-migrated accounts, the cloud-side data was deleted at the December 1, 2024 deadline (Google preserved at most the last 90 days as a one-time on-device courtesy). The Takeout path is therefore operationally dead for every user from December 1, 2024 forward. KML/KMZ variants of the same legacy data are rejected by the V1 parser with a "re-export as JSON" error — XML parsing plus KMZ inner-zip unpack adds dependency surface for no V1 benefit, and the JSON variants were always offered alongside.
  - **Device-side export (current path):** the Android Maps app exposes *Settings → Location → Location Services → Timeline → Export Timeline data*, which emits a single bare JSON file (typically `Timeline.json`, ~5–100 MB depending on retention window — the founder's three-month archive at the Settings.json `retentionWindowDays: 90` default is 22 MB / 908,234 lines). Top-level shape has three keys: `semanticSegments` (the semantic timeline — `visit`, `activity`, `timelinePath`, and `timelineMemory` entries; 1,386 total in the founder's archive distributed 48% / 28% / 24% / 0.2%); `rawSignals` (raw GPS positions, activity records, Wi-Fi scans); `userLocationProfile` (derived analytics like frequent-place HOME/WORK labels).
  - **First build (M16):** device-side export only. V1 ingests `visit` and `activity` entries from `semanticSegments` — 713 events from the founder's ~3-month Z Fold 6 archive (382 visits + 331 activities). V1 emits two distinct `event_type` values (`'visit'` and `'movement'`) under one `source = 'maps_timeline'`; payload shapes `MapsTimelineVisitPayload` and `MapsTimelineMovementPayload` in `packages/shared/src/schema/maps.ts`. Plugs into M15's `takeoutRegistry` via an `ingestArchive(unpackedRoot, sourceLabel, dataKey)` helper extracted from `ingestTakeout`: the bare-JSON upload affordance copies the user-selected `Timeline.json` into a fresh `codex-maps-` temp directory and invokes the same per-parser dispatch loop, so the Maps parser registers with the M15 registry exactly like the YouTube parser. The upload UX is a new `MapsSourceCard` distinct from the Takeout `.zip` uploader because the input shape is structurally different — a single bare JSON file rather than an archive needing unpack — and the user mental model is "your Maps export" rather than "your Takeout zip." See § 12.6 "Maps Timeline via device-side Android export" for the full architectural decision. V1.5: `timelinePath` ingestion (requires map rendering — 48% of the founder's archive segments are timeline paths that text-only rendering can't surface meaningfully), `timelineMemory` trip summaries (3 in the founder's archive — too sparse for V1 utility), `userLocationProfile.frequentPlaces` metadata for Sources card display, legacy pre-2024 Takeout format support if non-founder users surface preserved archives (see § 12.3). Never V1: `rawSignals` ingestion — Wi-Fi scan data inside `rawSignals.wifiScan.accessPoints[]` contains MAC addresses and signal strength of nearby APs at each location sample (uniquely de-anonymizing — an attacker with the same APs in scan range can prove the user was physically nearby), redundant with the semantic events the parser does ingest, and inconsistent with the encryption-first privacy story.

- **YouTube watch history**
  - **Live API:** none. The YouTube Data API does not expose a user's watch history (the legacy `watchHistory` activity feed has been retired). YouTube Music history is likewise gone from the Data API.
  - **Export:** Google Takeout, "YouTube and YouTube Music" → "history" → `watch-history.json`.
  - **First build:** export-only (no choice). Ingested via the generic Takeout pipeline. Replaces the early V1 placeholder of YouTube via Google OAuth.
  - **Export cap (rev 47 — M15 verification finding):** Google Takeout silently caps `watch-history.json` exports at ~10,000 entries. Observed on the founder's M15 verification archive: exactly 10,000 entries spanning 2023-11-03 22:28:28 UTC → 2026-05-09 20:06:25 UTC, with the oldest entry sitting at the bottom of the file (suggesting Google trims at export time on their side, before the JSON is generated, rather than truncating the file itself). No error from the export job, no warning in the JSON or the zip — the entries simply stop. Cross-checked via four independent signals on the uncompressed JSON: `"header":` regex hits, `"title":` regex hits, `"time":` regex hits, and `ConvertFrom-Json` array length all return exactly 10,000. The truncation is server-side, not in our pipeline (every entry that lands in the file gets ingested correctly), but it means future user archives with deeper history than ~10k watches will silently lose the older tail. A §12.3 V1.5 follow-up tracks the UX work of detecting the cap (`entries.length === 10000` flag) and surfacing a non-blocking notice on the source connection card so the user knows Google is limiting them rather than us. The `header` field on YouTube entries also has at least three distinct values in real archives — `"YouTube"`, `"YouTube Music"`, and `"YouTube TV"` — not just the two Google documents; the schema in `packages/shared/src/schema/youtube.ts` was broadened during M15 closing to reflect this. Real-world Takeout shapes routinely diverge from documented shapes, so future parsers (Maps Timeline at M16, etc.) should treat the documented shape as a starting point rather than a contract.

- **Photos with file content** *(promoted to Tier 1 in rev 30 — M14)*
  - **Live API:** Google Photos Library API supports `mediaItems.get` returning a `baseUrl` for download. Workable but rate-limited; ongoing token / refresh management; only covers photos in the user's Google Photos library. Currently metadata-only via the M3 Google integration.
  - **Export:** Google Takeout, "Google Photos" — full original-quality files in a zip.
  - **Content provider (Android):** `MediaStore.Images` and `MediaStore.Video` for on-device photos / videos — full file access, no cloud round-trip, but only covers media actually stored on the device.
  - **First build (M14):** Android `MediaStore` first. Thumbnail-first storage at 1024px max edge (~150KB per photo, ~95% storage reduction vs. full-resolution; see Section 12.6). Encrypted client-side in Kotlin via the per-blob HKDF-derived key pattern (Section 6 / 12.6), uploaded to Supabase Storage, with the encrypted-blob metadata landing in the new `event.blob_ref` jsonb column. Phase 1 ingestion (foreground, user-triggered "Sync gallery"); phase 4 background sync via `WorkManager` with Samsung battery-optimization handling. Full-resolution backfill, Takeout for desktop / Google Photos cloud library, and cross-source dedup across the M3 metadata-only feed and the M14 file-content feed are all V1.5 follow-ups (Section 12.3).
  - **Promotion rationale (rev 30):** photos are the highest-value source for the consumer artifact path (year-in-review, on-this-day, photo-rich digests). The Supabase Storage encryption pipeline that previously held this back is now the headline architectural work of M14, not a blocker.

## 12. Deferred Items & Known Issues

**Do not pre-emptively design for items in this section.** Fix when triggered, not before.

### 12.1 Bugs and Known Limitations

Things that work but aren't ideal. Each has a workaround or acceptable current behavior. Fix opportunistically.

- **Drive Activity actor identifiers:** Currently storing Google resource IDs (`people/12345`) in `participant_hash_set` instead of resolved email addresses. Won't correlate with email-based participants from Gmail/Calendar. *Workaround:* actors still appear by `displayName` in the UI; only cross-source person threading is affected. *Trigger to fix:* when building per-person threads, or when a user reports the issue.

- **Deep-link fallback on Windows:** The `flashback://` deep link scheme may not register reliably during `tauri dev` on Windows. *Workaround:* OAuth callback page includes a 3-second timeout and manual code-copy fallback; Sources screen includes a paste-code input. *Trigger to fix:* if deep links prove consistently unreliable in production, consider a polling-based approach.

- **Deep linking for password reset:** The reset link opens localhost in the browser rather than the app. *Workaround:* user can copy the reset URL manually. *Trigger to fix:* V1.5, when productionizing auth flows.

- **Zod version pin:** Using zod v4 with v3-compat imports as a workaround for `@hookform/resolvers` compatibility. Pinned to exact `4.4.3` in `apps/desktop/package.json` to prevent breakage. *Trigger to fix:* when `@hookform/resolvers` officially supports zod v4.

- **Rate limiting on auth forms:** Supabase has server-side rate limits but no client-side debouncing. *Workaround:* server-side limits are sufficient for solo use. *Trigger to fix:* before public launch.

- **Token refresh during ingestion:** V1 attempts a token refresh at the start of each ingestion call but does not handle expiry mid-ingestion. *Workaround:* most ingestion runs complete within token lifetime. *Trigger to fix:* when ingestion runs grow long enough to exceed token TTL regularly (V2).

- **`embed-events` Edge Function `verify_jwt` mismatch with docs:** Section 10 labels `embed-events` as "Service role" auth, but it's deployed with `verify_jwt: true` (any authenticated user JWT is accepted). The function source itself doesn't check the role. *Workaround:* the only callers in V1 are the desktop app (User JWT) and the auto-embed orchestrator (also User JWT), so the mismatch is benign today. *Trigger to fix:* when the embedding pipeline is touched next — either reconcile by setting `verify_jwt: false` and gating on the service-role key inside the function (matches Section 10), or update Section 10 to say "User JWT" (matches reality). Pick one and align both.

- **Post-save auto-scroll race uses fixed `setTimeout`:** Both the M5 search-result jump (`search-modal.tsx`) and the M9 manual-entry toast (`manual-entry-toast.tsx`) wait a fixed ~250ms after invalidating the timeline query before calling `scrollIntoView`, hoping React has re-rendered with the new row in the DOM. *Workaround:* the delay is generous enough in practice; misses are rare. *Trigger to fix:* when the timeline scroll polish pass is prioritized, or if a slow render makes the highlight miss visibly. *Migration path:* `await queryClient.invalidateQueries(...)` then `await new Promise(r => requestAnimationFrame(r))` before the scroll, instead of a magic-number timeout.

- **Ask Edge Function single-`{ question }` body backward-compat fallback:** `supabase/functions/ask/index.ts` still accepts a single `question` string in addition to the `messages[]` array introduced in rev 14 (M7). The desktop client only sends `messages[]`. *Workaround:* the fallback is harmless and adds <10 lines. *Trigger to fix:* once M7+ has run a few weeks without issue, drop the `question` branch and require `messages[]`; simplifies the contract.

- **Geist Mono OTS parsing warning at startup:** Chromium logs an OTS (OpenType Sanitizer) parse warning when loading `GeistMono-Variable.woff2`. Cosmetic only — the font still renders; Chromium falls back gracefully to system mono if it can't load. *Workaround:* the warning is silent to users. *Trigger to fix:* when overhauling typography or before any external release where DevTools console cleanliness matters; likely re-export the font with a more conservative naming-table or pull from a different distribution.

### 12.2 Performance Items

Acceptable now, will degrade at known thresholds.

- **FTS in-memory matching:** Currently fetches up to 1000 events client-side and matches HMACs in memory. *Trigger:* archive crosses ~1000 events OR search feels noticeably slow (>500ms perceived latency on a typical query). *Migration path:* change `encrypted_search_terms` column from `text` to `text[]`, add GIN index, move match logic into a Postgres RPC.

- **Vite build chunk size:** Build produces a chunk >500KB due to Supabase + crypto + React bundling. *Trigger:* bundle size visibly affects load time or app size grows past an unacceptable threshold. *Migration path:* code splitting with React.lazy and Vite manual chunks.

- **Ask retrieval window fixed at 30 events:** `TOP_K_EVENTS = 30` in `apps/desktop/src/lib/ask/conversation-limits.ts` is the cap on `match_events` results passed to Claude per turn. Adequate for narrow questions ("emails from Rebecca last May") but may miss relevant context for broad ones ("themes in my reading this year"). *Trigger:* user reports of "Claude doesn't seem to know about X" when X is in the archive. *Migration path:* dynamic K based on question type (cheap classifier on the question), or a hybrid retrieval (30 semantic + N keyword), or paginated retrieval with a "load more context" follow-up.

### 12.3 V1.5 Milestones

Real features we plan to ship. Not scheduled, but concrete enough to implement when prioritized.

- **OS keychain storage for data key (desktop):** Currently passphrase re-entry required on each app restart. Use Tauri's keyring plugin to store the data key in Windows Credential Manager / macOS Keychain / Linux Secret Service. User enters passphrase once per signup or after explicit lock. *Pairs with the Android sibling entry below ("Android Keystore storage for data key") as a cross-platform milestone covering both platforms' Kd-at-rest story; the threat model is shared (an attacker with disk access plus the OS-protected key-store secrets can recover Kd; an attacker with only filesystem access cannot) and the V1.5 work should land both halves together rather than letting one platform get keychain-backed durability while the other still requires passphrase re-entry on every cold start.*

- **Android Keystore storage for data key (V1.5 milestone, parity with desktop "OS keychain storage for data key" above):** The data key (Kd) is currently memory-only on both platforms per Section 7's V1 decision. This blocks background sync from running after OS kill / device reboot — `WorkManager` fires but `AuthSessionStore.current` is null, so the worker fails with `"Not signed in."` Surfaced during M14 Phase 4 investigation (Claude Code, post-rev 42). The natural Android implementation uses Android-Keystore-backed `EncryptedSharedPreferences` — already in use for the Supabase session via `EncryptedSessionManager`, so the infrastructure exists. Adding Kd to this store would require: (a) wrapping Kd at sign-in / unlock and storing the wrapped form; (b) unwrapping on app-process start when the encrypted session is restored, before any sync operation needs Kd; (c) zeroing Kd on signout including the wrapped copy. Threat model unchanged from desktop's keychain story: an attacker with disk access *and* the device's Keystore-protected keys can recover Kd; an attacker with only filesystem access cannot. *Trigger:* when M14 closes and the next-milestone selection revisits background sync's UX limitations. Naturally pairs with the desktop entry above as a cross-platform milestone covering both platforms' Kd-at-rest story. Phase 4 itself ships warm-process-only background sync (`WorkManager` runs while the process is alive; falls back to manual sync after OS kill) and explicitly defers the killed-process case to this future milestone.

- **Passphrase change flow (current passphrase known):** Re-wrap the existing data key with a new master key derived from the new passphrase. Distinct from password reset (which destroys data). Settings page has a disabled "Change passphrase" button with "Coming soon" tooltip — this work is trust-critical and deserves its own focused milestone.

- **AI provider strategy / BYOK:** V1 ships with single provider per capability — Voyage for embeddings, Anthropic Claude for Ask. Architect the LLM integration to be provider-pluggable so additional options (e.g., Grok) can be added without restructuring. User-selectable AI provider and bring-your-own-key (BYOK) pattern deferred to V1.5+.

- **"Remember me" toggle:** Supabase persists sessions in localStorage by default. A toggle to opt-out and clear session on window close is a possible future UX addition.

- **Persistent Ask conversations:** Conversations are memory-only — Zustand store is wiped on app restart, sign-out, or clearing the conversation. M7 noted this as a V1.1 deferral inline in the rev history but it never landed here as a concrete milestone. *V1.1 work:* persist `AskConversation` (with turns, citations, retrieved-context indices) to a new `ask_conversation` table behind RLS, encrypted client-side with the data key. Sidebar gets a conversation history list. Re-opening a conversation rehydrates the Zustand store.

- **Hilt DI migration:** M10 uses manual constructor injection for the small initial DI graph (~7 classes). Migrate to Hilt when graph exceeds ~15–20 classes or scoped lifecycles tied to navigation graphs / `ViewModelComponent` assistance are needed. Refactor milestone, not bundled with feature work.

- **Android launcher icon:** M10 shipped without an app icon. Manifest doesn't declare `android:icon`, so Android falls back to a generic system icon. Functional for solo dev. Required for any release build or sharing with users. *Trigger to fix:* when first sharing the app with someone other than the founder, or when releasing to Play Store. *What's needed:* source asset (SVG preferably); densities generated via Android Studio's Image Asset Studio. Branded icon design itself is a separate prerequisite to having an asset.

- **Phone-side edit/delete for manual entries:** M12 ships read-only manual entries on Android — `ManualEntryDetailView` has no edit or delete affordance. Edit requires form UI + write-and-re-encrypt + auto-embed re-trigger; delete requires a confirmation dialog matching desktop's pattern (M9). Both are real chunks of work that don't belong bundled into "view detail." Manual entries remain editable from desktop in the meantime. *Trigger to fix:* when phone-only authoring becomes a real workflow (e.g. user finds themselves wanting to capture a memory on the go and edit it later from the same surface). *Migration path:* `ManualEntryEditor` composable with autoexpanding textarea + native date picker, `EditManualEntryViewModel` driving the encrypt-and-update path, then a separate confirm-and-delete flow. Land both together — partial parity (edit but no delete, or vice versa) is more confusing than no edit.

- **Hard-delete for shared artifacts:** V1 sharing is revoke-only — clicking "Unpublish" sets `shared_artifact.revoked_at = now()` and the row stops resolving via the anon-read RLS policy, but the cleartext payload remains in our database. Hard-delete (deleting the `shared_artifact` row entirely) is needed before any users beyond the founder are onboarded to sharing — revoke alone violates user expectation that "I can take it down completely." *Trigger to fix:* before any non-founder user uses the public-sharing feature. *Migration path:* add a "Permanently delete" affordance distinct from "Unpublish" in the published-artifacts list; the row is `delete`d (not just updated) and the slug becomes available for reuse. Cloudflare edge caches need an explicit purge call alongside the delete to avoid serving the cached cleartext after deletion. Tier 2 follow-up tied to the public-sharing milestones.

- **Direct posting from Codex to social platforms (Tier 2+):** Users will eventually want to publish generated artifacts (collages, digests, charts) directly to their social accounts rather than just to a Codex public URL. This is *outbound* publishing across many platforms, with all the per-platform OAuth, format, and rate-limit complexity that implies — distinct from the inbound ingestion landscape captured in Section 11. *Open architectural decision when this milestone surfaces:* build the integration layer ourselves, or use a service like Outstand (or similar cross-platform publisher) that's purpose-built for this. *Trigger to fix:* when Tier 2 sharing has shipped and at least one user has explicitly asked for direct platform posting.

- **View-in-source intent deep links from event detail:** Tapping a Gmail / Calendar / Drive / Spotify event detail could open the source app at the original message / event / file / track. Both platforms have a path: desktop opens a URL (`mailto:`, web URL, or custom scheme) the OS hands to the registered app; Android uses an `Intent` against the same schemes. The V1.5 work covers whichever platform surfaces the friction first. Each source has its own intent schema (Gmail uses `mailto:` plus message-id queries, Calendar uses `content://` event URIs with calendar IDs, Drive uses Drive web URLs intercepted by the Drive app, Spotify uses `spotify:track:...` URIs). Some apps (Drive Activity in particular) don't deep-link cleanly to specific actions. Separate polish milestone after the basic detail surface stabilizes; worth doing per-source rather than building a generic "open in source" abstraction. *Trigger to fix:* when the user reports wanting to "go to the actual email" enough to be a real friction point.

- **Per-source "Reset and re-sync" affordance:** M13 has no in-app way to clear a single source's ingestion cursor and re-ingest from scratch without wiping all app data (which also blows away the auth session, requiring re-derivation of the master key). Surfaced during M13 verification: a date-format change in the SMS payload (preferring `Telephony.Sms.DATE_SENT` over `DATE` so restored messages keep their original timestamps) required wiping all SMS rows in Supabase *plus* clearing the app's storage to reset the per-device cursor in `EncryptedSharedPreferences`. *Migration path:* add a "Reset this source" action on each source card that clears that source's cursor and event count without affecting auth or other sources. *Trigger to fix:* V1.1 polish, or sooner if a non-founder user encounters the same scenario.

- **Background sync UX polish (M14 follow-up):** M14 phase 4 adds `WorkManager`-based background sync with Samsung battery-optimization handling — *infrastructure* lands in M14, replacing the previously-tracked rev-28 "dedicated background sync milestone" plan (continuous capture is now part of M14's "data density" gate rather than a separate later milestone). What remains as polish: user-facing nudges if Samsung's battery optimization restricts Codex (the OEM-specific permission flow is a non-trivial UX ask, and silent failure is the worst case); visual indicators of sync progress when the app is foregrounded after background work completed or partially completed; graceful handling of mid-sync interruptions (network drop, OS-imposed sleep) without user-visible error states. V1.1 polish, scoped after M14 ships and real usage surfaces which polish actually matters.

- **Contact resolution for SMS cards (V1.1 polish):** SMS events display phone numbers, not contact names. Resolving requires the `READ_CONTACTS` permission and a contact-lookup pipeline (and graceful handling of contacts that change or are removed after the SMS row was ingested). Polish, not core — the underlying message data is intact regardless of whether a name is shown.

- **Cross-platform line-ending policy (`.gitattributes`):** The project doesn't currently have a `.gitattributes` file specifying line-ending normalization. On Windows machines with `core.autocrlf=true`, files saved by editors using LF endings (default in some VS Code / Cursor configurations) appear as "modified" in `git status` even when content is byte-identical to the committed version. Surfaced during M13 push: `apps/desktop/src-tauri/Cargo.toml` flagged as modified on a Windows machine despite no actual content change. Resolution then was a one-off `git checkout`. Long-term fix: add `.gitattributes` with explicit eol policies per file type (probably `text=auto eol=lf` for source files, with exceptions for `*.bat` / `*.ps1`). *Trigger to fix:* when a second contributor (or second machine) hits the same false-positive pattern, or before the project starts accepting external contributions.

- **Cross-source photo dedup (M14 follow-up):** M14 ingests from the Android gallery only. Future milestones may add Google Photos with file content (currently metadata-only via the M3 OAuth integration) and other photo sources (iOS camera roll once iOS lands; Takeout-based bulk imports). The `dedupe_key` derivation in M14 uses (capture timestamp + dimensions + filename) which *should* produce stable identity across sources for the same underlying photo, but collision behavior between Android-gallery-photo and Google-Photos-with-file-content has not been verified because both source variants don't yet exist concurrently. *Trigger to fix:* when adding a second photo source. *What's needed:* sample data from both sources for the same underlying photo, dedup heuristic validation, fallback strategy for sources that don't expose comparable metadata (e.g. some Takeout exports lose original filenames).

- **Full-resolution photo backfill (M14 follow-up):** M14 stores thumbnails only (1024px max edge, ~150KB per photo — see Section 12.6). Full-resolution backfill is a future operation, triggered when the user explicitly requests high-resolution access — generating a year-in-review with print-quality output, downloading the original from event detail, or sharing a high-res copy publicly. Architecture should accommodate this without a schema migration: the `event.blob_ref` jsonb column (Section 6) can hold both a thumbnail variant and a full-resolution variant under named keys (e.g. `thumbnail` + `full`), with the per-blob HKDF derivation parameterized by variant. No V1 implementation; documented now so the M14 schema doesn't paint into a corner.

- **True streaming AES-GCM for blob encryption (V2 upgrade path, M14 phase 1 follow-up):** V1 uses an API-streaming surface backed by single-shot AES-GCM encryption (Section 6); the full plaintext is held in memory during encrypt and decrypt. This is acceptable for V1's thumbnail-first blob sizes (~150KB–5MB per photo). The Rust ecosystem currently lacks an in-tree streaming AES-GCM API in the `aes-gcm` crate (RustCrypto/AEADs#556 declined to add one on the grounds that buffering complicates one-shot performance optimization); `aes-gcm-stream` exists as a thinner wrapper exposing single-nonce + trailing-tag streaming but adds dependency surface that we declined for V1. *Trigger to revisit:* when video ingestion, full-resolution photo backfill (the previous follow-up entry), or multi-GB blobs surface as concrete requirements. *Migration path:* swap the internal encryption call in `blob_aead.rs` from `Aes256Gcm::encrypt(accumulated_buffer)` to a true-streaming implementation (either `aes-gcm-stream` or a manual `aes` + `ghash` integration); the wire format does not change because single-nonce + trailing-tag is exactly what a true-streaming implementation would emit. Existing blobs do not need to be re-encrypted. The chunked API surface (`BlobEncryptor::encrypt_chunk` / `finalize`) does not change. Kotlin side is already true-streaming via JDK `Cipher.update()` / `doFinal()`, so the upgrade is Rust-side only.

- **Cross-platform `crypto-vectors.json` file size (M14 phase 1 decision, accepted):** The shared test vector file (`packages/shared/test-vectors/crypto-vectors.json`) grew from ~4KB to ~4.2MB on the M14 phase 1 commit because the `streaming_aes_gcm` group includes a real-photo-shaped 1MB plaintext vector (`one-mb-plaintext-recommended-chunks`) — the plaintext encodes to ~2MB of hex and the corresponding ciphertext another ~2MB. Decision: **accept the file size**. The 1MB vector is the only one that exercises the chunked API at the canonical production chunk size (`RECOMMENDED_CHUNK_SIZE` = 64KB) with a meaningful number of chunks (16). Smaller representative vectors do not catch chunking bugs that only surface at multi-chunk scale. Alternatives considered and declined: (a) replace with a 200KB vector — would only exercise 4 chunks, weaker coverage; (b) split the large vector into a sidecar file (`crypto-vectors-large.json`) — adds asymmetric loader complexity on both Rust and Kotlin for marginal benefit; (c) deterministic-seed regeneration on each test run — moves complexity from the JSON to the verifier on both platforms and breaks the "JSON is the contract" invariant from M10. Fixed file size; does not grow with future vectors unless we add another large-plaintext case. *Trigger to revisit:* if the repo becomes sensitive to large blobs in git history (e.g., before adding a CDN-distributed vector set), or if a third platform is added and the parsing cost of the 1MB hex string becomes a concern.

- **Orphan blob cleanup (M14 phase 2 follow-up; figures empirically corrected rev 45):** During M14 phase 2 verification a PostgREST 1000-row response cap silently truncated the gallery ingestor's client-side dedup query (the cap is the default `X-Max-Rows` on Supabase REST), causing repeated re-encrypt + re-upload of already-ingested photos at fresh `blob_uuid`s. The event-table unique constraint on `(user_id, source, source_event_id)` rejected the duplicate row inserts, but the encrypted blobs DID land in Storage with no event row pointing at them — orphans. Empirically measured during M14 Phase 5 step A4 (read-only `storage.objects` ∖ `event.blob_ref` diff on 2026-05-11): **607 orphan blobs totalling 63.9 MB** accumulated on the founder's verification archive — 1961 storage objects under the founder prefix in the `blobs` bucket versus 1354 distinct `event.blob_ref ->> 'storage_path'` values for `source = 'android_gallery'`. All 607 orphans were created in a single ~46-minute window on 2026-05-10 between 15:47:50 and 16:33:45 UTC during the initial archive upload sequence — temporally bounded, not ongoing leakage. The orphan paths are well-formed canonical uploads (`<user_uuid>/<random_uuid>` shape, size range ~10 KB to ~381 KB with mean ~108 KB, matching the expected thumbnail-blob shape from Section 12.6) that never got paired event rows, not malformed garbage. The earlier "~250–400 orphan blobs (~50 MB)" figure in this entry was an estimate written from architecture guidance before the actual orphan set was enumerated; the rev 45 measurement corrects it to the empirical 607 blobs / 63.9 MB. **Phase 5 A4 also confirmed the bug is closed:** Phase 4 verification day (2026-05-11) added exactly 2 storage objects, both paired with matching event rows, so the entire 607-blob mass is attributable to the bounded Phase 2 incident on 2026-05-10. The PostgREST cap was fixed in the M14 phase 2 commit by paginating the dedup query, so no new orphans are produced. Cleanup of the existing orphans, and a defensive idempotent cleanup pass for any future bug-shaped-like-bug-2 incidents, lands as its own small standalone milestone after M14 phase 5 rather than getting bundled with the account-deletion-cascade work that's already on the V1.5 list. *Identification:* enumerate `storage.objects` under each user's prefix in the `blobs` bucket and exclude paths present in any `event.blob_ref ->> 'storage_path'` — the difference is the orphan set. *Risk if deferred:* compounds — every future bug-shaped-like-bug-2 at scale adds more orphans; per-user storage cost grows; and the cleanup query gets slower as the orphan set grows. *Trigger:* end of M14 phase 5 (full archive ingest will surface any remaining bug-2-shaped paths), or sooner if Storage cost on the founder account exceeds the free tier.

- **Prompt templates as data objects with versioning and permissions (Tier 2 implementation note, V2+ enabling):** When Tier 2's first signature artifact (year-in-review) is implemented, the prompt templates that drive generation should be stored as **data objects** with proper schema support for versioning, attribution, and permission scopes — even though Tier 2's templates will be first-party Codex-built only. *Required schema shape* (for migration when this work begins):

    ```text
    prompt_template:
      id                     uuid
      version                int
      author_user_id         uuid, nullable        -- null means 'system'
      parent_template_id     uuid, nullable        -- for forks
      name                   text
      description            text
      category               text                  -- 'year_in_review', 'brief_me', etc.
      prompt_body            text                  -- prompt template with interpolation variables
      output_schema          jsonb                 -- generated artifact's data shape
      rendering_template_id  uuid, nullable        -- linked designed rendering template (separate from prompt body)
      permissions            jsonb                 -- '{visibility: private|shared|public, shared_with: [user_ids]}'
      created_at             timestamptz
      updated_at             timestamptz
    ```

    This shape supports Tier 2 first-party use (`author_user_id = null`, `visibility = 'private'`, everyone uses the same templates) **while** enabling V2+ community-template-sharing (`author_user_id = user_id`, `parent_template_id = fork-source`, `visibility = 'public'`) without schema migration. *Why land this at Tier 2 implementation rather than V2+:* retrofitting versioning and permissions onto a prompts-as-code implementation requires both schema migration *and* restructuring how generation engines consume templates. Designing the schema correctly from the first template implementation is small additional work; the architectural cost of getting it wrong is high, the cost of getting it right early is negligible. *Trigger:* M18-or-whenever year-in-review architecture is designed in Tier 2. This entry exists so the design includes the schema shape rather than treating prompts as hardcoded strings. See § 12.4 "Codex as personal context substrate with agent generation surface" for the V2+ direction this enables.

- **CSP hardening (V1.5 follow-up):** `apps/desktop/src-tauri/tauri.conf.json` currently has `"csp": null` — Content Security Policy disabled. Acceptable while the desktop UI was text-only and no external content was loaded. M14 Phase 3 introduces Supabase Storage reads (encrypted blob bytes fetched into the webview) which expands the surface where a CSP would matter. Future work: define a tight CSP that allows Supabase Storage domains for blob fetches and the Anthropic API endpoint for Ask, blocks everything else, and disables inline scripts beyond what Tauri's bundle requires. Wildcard `connect-src` is the wrong shape; an explicit allowlist is. *Trigger:* when any external content beyond Supabase + Anthropic API is loaded by the desktop app, or when a security-pass review is scheduled for V1 launch.

- **Duplicate `QueryClientProvider` in desktop app (V1.5 follow-up):** Surfaced during M14 Phase 3 desktop investigation. There are two `QueryClientProvider` instances wrapping the app — one in `apps/desktop/src/main.tsx` (line 11) and one in `apps/desktop/src/App.tsx` (line 71) — each instantiating its own `new QueryClient()`. The inner provider in `App.tsx` shadows the outer one in `main.tsx`; components only see the inner client. **Functionally not breaking** (queries still work because the inner client handles them), but means TanStack Query devtools point at the wrong client, query invalidation from outside the React tree (e.g., an imperative `queryClient.invalidateQueries()` call from a Zustand action) silently uses the wrong client, and any future cross-context query coordination misbehaves. *Fix:* one-line — remove the wrapper in `App.tsx`, keep `main.tsx` as the single source. *Trigger:* anytime, low priority — bundle with whatever next desktop refactor touches `App.tsx`.

- **Tauri version pin (V1.5 follow-up):** `apps/desktop/src-tauri/Cargo.toml` currently has `tauri = { version = "2", features = [] }` which floats across all 2.x minor versions (Cargo's caret-equivalent semantics for bare-major version strings). Pinning to a specific minor (e.g., `"2.4"` or whatever's current at fix time) would be defensive against minor-version surprises in Tauri's filesystem, path, and command APIs — particularly relevant as Phase 3 introduces new Rust commands that touch `app_local_data_dir()` and `std::fs`. Not blocking; M14 Phase 3 works fine with the floating version. *Trigger:* anytime, low priority — bundle with whatever next desktop change touches `Cargo.toml`.

- **Base64 transport overhead for blob IPC on desktop (M14 Phase 3 implementation finding, V2 revisit):** The encrypted-blob path on desktop crosses the JS↔Rust IPC boundary as base64 strings (the established pattern for crypto commands). For 150KB thumbnail blobs, this is ~200KB base64-encoded plus a UTF-16 string allocation in V8 — measurable but acceptable. For the future full-resolution path (~5MB blobs), this would be ~6.7MB strings per blob, which is worth revisiting. Two known forward paths: `tauri::ipc::Response` with raw bytes (lower-overhead IPC return), or the custom Tauri protocol already documented in the rev 40 § 12.6 entry for blob URL transport. Phase 3 already uses `tauri::ipc::Response` for the *return* of `decrypt_blob_bytes_and_cache` and `read_cached_blob` (raw `ArrayBuffer` to JS, no base64-on-the-way-back tax); only the encrypted-bytes upload from JS → Rust still uses base64. Android side doesn't have this overhead because Coil's Fetcher takes bytes directly from the Kotlin Supabase client to the Kotlin `BlobDecryptor` — no bridge crossing. *Trigger to revisit:* when full-resolution rendering ships (separate V1.5 follow-up) or when desktop IPC overhead becomes measurable in production.

- **Detect Takeout export caps and surface to user (M15 follow-up, rev 47):** Google Takeout silently caps `watch-history.json` exports at ~10,000 entries (see Section 11 "YouTube watch history" mini-spec for the empirical evidence from the founder's M15 verification archive). The cap is opaque — no error from the export job, no warning in the JSON or zip, no metadata field that says "we trimmed this." Future user archives with deeper history than the cap will silently lose the older tail. *V1.5 work:* detect the cap heuristically (`entries.length === 10000` for YouTube watch history; analogous detection for other Takeout categories once their caps surface) and surface a non-blocking notice on the source connection card in the Sources screen reading something like "Google's export limited this archive to 10,000 entries. Older history may be missing." The cap is server-side, not in our pipeline — every entry that lands in the file gets ingested correctly — so this is purely UX work, not pipeline work. *Trigger:* anytime a non-founder user uploads a Takeout archive deeper than the cap and we want them to know Google trimmed it rather than wondering why their history starts in late 2023.

- **Cursor pagination tiebreaker for timeline infinite scroll (M15 follow-up, rev 47):** `apps/desktop/src/components/timeline/timeline.tsx` paginates with `query.lt('occurred_at', pageParam)` (strict-less-than) using the previous page's last `occurred_at` as the cursor. Without a secondary tiebreaker, pages silently skip overflow when 51+ events (a full `PAGE_SIZE` plus one) share the exact same `occurred_at` timestamp — the next page's `lt(T)` clause excludes all events at time T, even the ones not yet shown. YouTube watch history routinely has multiple entries at the same `time` ISO string (rapid clicks, autoplay), so the latent bug is real but rare. *Fix:* add a secondary cursor on `id` so pagination becomes `(occurred_at, id) < (lastOccurredAt, lastId)` lexicographically; PostgREST supports this via `or(...)` with `and(...)` nested predicates, or a Postgres function wrapping the comparison. *Trigger:* anytime a heavy user surfaces the scenario (51+ events sharing a `time` ISO string at second precision — burst-watching during a YouTube binge, or a different source surfacing sub-second clustering), or measurement during M15 closing's full-archive scroll verification shows skipped events.

- **Virtualization for the timeline at scale (M15 follow-up, rev 47):** `DaySection` in `apps/desktop/src/components/timeline/day-section.tsx` does `events.map(...)` with no virtualization library — every `EventCard` lives in the React tree at once. Fine at <2k events (M14 Phase 3 scale; the founder's pre-M15 archive at ~2k events scrolled smoothly), expensive at the M15-scale 11.4k+ events on the founder's verification archive (10k YouTube watch history + 1.3k Android gallery + ~100 misc): framer-motion's per-card animation `delay: index * 0.04` queue saturates, React reconciliation cost spikes during scroll, the layout pass over thousands of DOM nodes hits the main thread. Latent on the founder's current hardware (Apple Silicon / desktop-class CPU), worth a real measurement-driven milestone when archives routinely hit 30k+ events or when scroll jank surfaces in production. *Fix:* drop in `@tanstack/react-virtual` (zero-runtime, hook-based, integrates cleanly with the existing `useInfiniteQuery` pattern) and virtualize at the `DaySection` level — render only the visible day sections plus a small overscan window, with measured-row sizing because event cards have heterogeneous heights (Gmail subject lines + snippets are taller than YouTube watch rows). *Trigger:* user reports scroll jank at scale, or measurement on a real archive shows >100ms React reconciliation cost per frame during scroll. Separate from the cursor tiebreaker entry above — different latent issue, different fix shape.

- **Legacy pre-2024 Takeout Maps Timeline support (M16 follow-up, rev 48):** M16 ingests Maps Timeline via the post-2024 Android device-side export format only (single `Timeline.json` with `semanticSegments` / `rawSignals` / `userLocationProfile` top-level keys). Pre-2024 Google Takeout exports (when Location History still lived in the cloud) used two earlier formats: `Takeout/Location History (Timeline)/Records.json` (a single flat array of raw GPS samples under `{ "locations": [...] }`, hundreds to thousands of entries per day per user; per-record fields include `timestamp` ISO-8601, `latitudeE7` / `longitudeE7` integer coordinates, `accuracy`, `velocity`, `heading`, `altitude`, `source` (`"WIFI"` / `"CELL"` / `"GPS"`), `placeId` when associated with a known place, `platformType`, `deviceTag`, hierarchical `activity[]` arrays per Android's `DetectedActivity` constants); and `Takeout/Location History (Timeline)/Semantic Location History/<year>/<year>_<MONTH>.json` (per-month files under `{ "timelineObjects": [PlaceVisit | ActivitySegment, ...] }`, the closest legacy analogue to the modern `semanticSegments` shape — `placeVisit` has `{ location: { latitudeE7, longitudeE7, placeId, name, address }, duration: { startTimestamp, endTimestamp }, visitConfidence, ... }`; `activitySegment` has `{ startLocation, endLocation, duration, distance, activityType, confidence, ... }`). When non-founder users surface preserved pre-2024 Takeout archives containing either legacy format, add multi-format support to the Maps parser: top-level shape detection (`{ "locations": [...] }` → legacy Records, `{ "timelineObjects": [...] }` → legacy Semantic, `{ "semanticSegments": [...], "rawSignals": [...] }` → device-side current) with three separate field-mapping branches, all converging on the same `MapsTimelineVisitPayload` and `MapsTimelineMovementPayload` shapes so the downstream UI surfaces don't fork. V1 ships device-side only because the founder's archive is post-migration (the December 2024 cloud-data deletion left non-migrators with nothing and migrators with device-side data); legacy support waits for demonstrated demand from a real user with a preserved archive. *Trigger:* anytime a non-founder user attempts to upload a pre-2024 Takeout zip and surfaces the format mismatch, or when historical-data import becomes a stated user need.

- **Maps Timeline `timelinePath` ingestion for map rendering (M16 follow-up, rev 48):** 670 of 1,386 `semanticSegments` (48%) in the founder's M16 verification archive are `timelinePath` entries — sequences of GPS points emitted in 2-hour windows between visits and activities, representing the continuous track of where the device travelled. - **User unit preference — metric / imperial (Day 2 dark-pivot follow-up, rev 50):** V1 demo hardcodes US imperial units in the Maps Timeline movement card and detail panel — `formatDistance` in `apps/desktop/src/components/timeline/event-card-maps-movement.tsx` returns `"X.X mi"` (conversion `meters / 1609.344`), and the average-speed display in `MapsMovementDetail` inside `apps/desktop/src/components/timeline/event-detail-panel.tsx` returns `"X mph"` (conversion `m/s · 2.236936`). Hardcoded imperial is the right call for the V1 demo because the founder is US-based and the Speedrun submission audience is US-anchored, but it's the wrong long-term default — any user outside the US will read `12.3 mi` as friction, and we should surface the unit system as a user preference rather than baking US-centric assumptions into the data layer. *Two clean migration paths for V1.5:* (a) **automatic inference from `navigator.language`** — at app boot, parse the BCP 47 region subtag (e.g., `en-US` / `en-GB` / `de-DE`) and pick `imperial` for US / Liberia / Myanmar and `metric` for everywhere else; expose the inferred value as a `useSettingsStore` selector and have `formatDistance` / `formatSpeed` consume it. Cheap, defaults sensibly for 95%+ of users, no UI surface required. (b) **explicit Settings preference** in the Day 3+ rebuilt Settings page (Units section: radio `Imperial (mi, mph, °F)` / `Metric (km, km/h, °C)` / `Auto` defaulting to `navigator.language` inference); the explicit choice overrides the inferred value and persists in the Zustand-backed settings store + the user's encrypted profile so it follows them across devices. (b) is the more user-respecting answer; (a) is the cheaper interim. Both require touching `formatDistance` / `formatSpeed` to read from the settings store rather than hardcoded imperial, so the migration is a 5-line change at the helper layer plus the settings UI work itself. *Trigger:* the first non-US user joining (the second user, in practice; the first will likely be a US-based investor or beta tester) or any time before public launch — whichever comes first. The Maps Timeline card and the Maps Movement Detail are the only V1 surfaces displaying distances or speeds; the broader unit-preference scope (Fahrenheit / Celsius for any future weather data, kilograms / pounds for any health-tracking surfaces, etc.) is an architectural placeholder for whenever those surfaces actually exist.

### 12.4 V2+ Architectural Direction

Big-picture decisions about future versions. Not concrete roadmap — directional only.

- **Local ONNX embeddings:** Move from Voyage to on-device embedding generation. Eliminates third-party data transit for semantic search.

- **Embedding encryption at rest:** V1 stores embeddings unencrypted (documented in `ENCRYPTION.md`). Move to client-side encrypted embeddings or a privacy-preserving similarity scheme.

- **Cross-device session sync:** V1 supports multiple desktops; user enters passphrase on each. No "trusted device" sync of passphrase. V2 consideration.

- **Export format standardization:** V1 uses JSON-based archive of decrypted events (downloaded by client). Standard format TBD for V2.

- **Pricing model:** Subscription assumed, exact tier and price TBD. Build the product first.

- **Marketing site:** Out of scope for V1 build; single Vercel-deployed Next.js page later.

- **Manual entry `updatedAt` lives in encrypted payload, not as a column:** M9 chose to store `ManualEntryPayload.updatedAt` inside the encrypted JSON rather than add an `event.updated_at` column with a trigger. Keeps `updatedAt` encrypted, avoids a one-off migration, but means the timestamp is invisible to server-side queries. *Trigger to promote to a column:* when any feature needs to query/sort/filter by update time server-side (e.g. "show recently edited entries first" in the Notes view, or a sync-since-X API for a future mobile app). *Migration path:* add `updated_at timestamptz default now()` to `event` with an `on update` trigger; backfill from payload-embedded values for manual entries; treat as optional/null for other event types where the concept doesn't apply.

- **Codex as personal context substrate with agent generation surface (V2+ direction):** The platform-shape direction Codex is heading toward, articulated for architectural alignment even though the work is years away.

    *Underneath:* an encrypted, normalized, entity-aware archive (the substrate) that's continuously enriched through ingestion and indexing. The substrate's job is to make *useful programs possible against the user's context*, not to be the product itself.

    *On top:* a generation surface that composes agents — both templated and ad-hoc — against the substrate. Artifacts (year-in-review, brief-me, on-this-day, relationship retrospective, project postmortem) and persistent agents (weekly digest, deadline monitor, project tracker) are surface variations of the same underlying primitive: a prompt + schedule + scope that runs against the user's context and produces a designed output. The **artifact-as-agent collapse** is a unifying architectural insight — these are not different feature categories but different parameter shapes against the same generation engine.

    *Templates are the consumer wedge; open composition is the power-user surface.* Tier 2's first signature artifacts (year-in-review with Felton-style report; subsequent named templates) are the first instances of this architecture. Generation Workspace exposes the same architecture as a creative surface where users describe arbitrary jobs and the platform composes them.

    *Eventual platform shape (V2+):* scoped third-party access via user authorization, similar to OAuth-mediated app permissions, where Codex remains the consumer-facing substrate that owns the user relationship and external apps integrate as authorized consumers of context. The substrate's encryption boundary supports this through **per-recipient share keys** (M14's per-blob HKDF derivation pattern is the prototype for per-share key derivation; Section 7's key hierarchy already includes Kb derived from Kd via HKDF-SHA256 with structured `info` parameters, and a future Kshare can follow the same construction with `info = 'codex.share' || share_uuid`). Third parties never decrypt the user's archive directly; they request scoped views the user authorizes, and the user's client decrypts and shares only what was authorized.

    This is not a Tier 1 / Tier 2 commitment. It is a direction the architecture should not preclude. *Specific architectural implications worth registering now:*

    - **Prompt templates should be stored as data objects with versioning, attribution, and permission scopes from initial implementation,** even when only first-party Codex templates exist. Schema designed for community-shareable templates from day one is small work; retrofitting versioning and permissions later is large work. See § 12.3 follow-up entry "Prompt templates as data objects with versioning and permissions" for the concrete schema shape.

    - **The generation surface should treat artifacts and persistent agents as the same primitive.** A "year-in-review" is structurally a once-yearly persistent agent. A "weekly digest" is a weekly persistent agent. A "brief me" is an on-demand agent. The schedule parameter varies; the engine doesn't.

    - **Scoped access for future third-party integration should reuse the principle from public sharing (rev 26 decision):** cleartext exits the encryption envelope only with explicit user action, with the scope and recipient specified at the time of share / grant.

    The Context Revolution framing in the broader AI ecosystem suggests this direction is becoming a standard architectural shape; the Codex implementation route — encryption-first, consumer-positioned, ingestion-broad — is one of the few that can credibly own the consumer-substrate position. Architectural decisions made today should not preclude this evolution.

Block public launch but don't matter for solo developer use.

- **Spotify Extended Quota Mode:** Currently in Development Mode (requires Premium on developer account, max 25 test users). Ingestion code is complete and tested; this is purely a platform-level gate. Required before public launch.

The founder can use the app safely (he scheduled it; he knows), but external users have no out-of-band signal — if their account is compromised and an attacker schedules deletion, they'd only learn by signing back in within 7 days. *Trigger to fix:* before any non-founder user signs up. *Migration path:* call Supabase Auth's transactional email or a separate provider from the schedule path; mirror message in the `cancel` path.

### 12.6 Architectural Decisions (decided but worth noting)

Choices that previously sat in 12.1 as "deferred" but have since been resolved. Recorded here so future-us doesn't re-litigate them and so the rationale is preserved.

- **Two implementations of crypto, kept in parity by shared test vectors:** Rust on desktop (via Tauri), Kotlin on Android (Tink + Bouncy Castle). Single shared schema/tests do not exist because TS/Rust/Kotlin can't share native-typed crypto code cleanly. Parity is enforced instead by `packages/shared/test-vectors/` — a JSON file of input → expected-output triples that both `cargo test` (desktop Rust) and Gradle's `:app:test` (Android Kotlin) load and verify against. Any change to crypto on either platform must (a) update the vectors and (b) pass the verification tests on both platforms before merge. Decided rev 19. Replaces the previous 12.1 entry that flagged `packages/shared/src/crypto/` as Tauri-coupled and asked for a "platform interface" abstraction — that abstraction is unnecessary because Android stands up its own native implementation rather than sharing TS code at all.

- **AES-GCM via JDK Cipher (not Tink):** The `.cursorrules` and Section 4.5 originally specified Tink for AEAD. Tink's `AesGcmJce.encrypt` always generates a random nonce with no public override, which prevents deterministic ciphertext generation needed for cross-platform parity vectors. JDK `Cipher` (with explicit `GCMParameterSpec`) is used instead. The underlying primitive (AES-GCM) is identical; only the API surface differs. Decided rev 20.

- **Supabase Kotlin SDK pinned at 2.6.1 (not 3.x):** The 3.x line renamed `gotrue-kt` → `auth-kt` and bumped major version with breaking changes. M10 was scoped against the 2.x API. Staying on 2.6.1 (latest 2.x stable) avoids the rewrite. Migration to 3.x is a follow-up milestone when 3.x has fully stabilized and we have explicit reason to upgrade. Decided rev 20.

- **Engine first, platforms as delivery mechanisms (rev 26):** The product is the engine — data model, ingestion pipeline, reasoning logic, generation logic, sharing infrastructure. Platforms (Tauri desktop, Android Kotlin, future iOS, future web) are *delivery mechanisms* for the same engine. Most product logic lives server-side (Edge Functions, schema) or in shared client logic. Platform-specific code is for rendering and platform-native capture (e.g. SMS via Android content provider). New platforms should be cheap to add precisely because the engine doesn't move when a new client lands. This decision shapes how new milestones are scoped — a feature is built engine-first, then exposed per platform — and it's why iOS is V2-but-architecture-must-not-block-it rather than V2-rewrite. Encoded as principle 7 (Section 2). Decided rev 26.

- **Public sharing as deliberate cleartext exit from the encryption envelope (rev 26):** Public sharing (Tier 2+) is opt-in per entry; shared content is *cleartext* in the new `shared_artifact` table (Section 6) so anonymous viewers can read it without a key, while the user's private archive remains encrypted client-side as before. Two coexisting guarantees: "only you can read your private archive" (unchanged) and "shared content is exactly what you chose to share, no more" (new). Generated `artifact` rows remain encrypted by default even though the user might share some later — drafts must stay private (the user generates ten briefing variants, publishes one; the other nine are not exposed). Documented in Section 7's "Public sharing and the encryption boundary." Decided rev 26.

- **Cloudflare Pages for the public surface (rev 26):** Public shared artifacts are rendered on a separate `apps/public/` codebase deployed on Cloudflare Pages, served from its own subdomain (TBD: `share.flashback.app` or similar). Static-first hosting with minimal JS; reads `shared_artifact` directly via the anon-RLS policy rather than going through any custom API. Privacy-aligned (Cloudflare's business model isn't ad-supported), scales to free tier well past our needs, simple deploy from git, edge-cached for performance. Astro or plain HTML+CSS preferred over a heavy React framework — the public surface should not share a component library with desktop. Decided rev 26.

- **Live-API-first with export-zip fallback (rev 27):** Most platforms have closed or restricted their data APIs for third-party consumption while still honoring export rights under GDPR-style portability laws. Codex's ingestion strategy is: live API where stable and reasonably accessible (current: Google services, Spotify), export-zip upload for the rest. Per-platform may eventually support both paths — live for first-impression UX, export for completeness. Build order per platform is decided per milestone; we don't pre-build a unified plugin pattern speculatively (concrete examples will yield a better abstraction than imagined ones — principle 6). The per-source landscape lives in Section 11; the principle is encoded as principle 11 (Section 2). Decided rev 27.

- **Per-blob derived encryption key via HKDF (M14, rev 30):** Photos and other binary blobs use a per-blob key derived from the user's data key Kd via HKDF-SHA256, with the blob UUID as part of the HKDF `info` parameter (`'codex.blob' || blob_uuid`). Eliminates the AES-GCM nonce-reuse failure class structurally — two blobs sharing a nonce under the same key would compromise both, and per-blob keys make that scenario impossible by construction rather than by careful nonce bookkeeping. HKDF cost is essentially zero (already in the crypto facades from M2; the search key Ks is derived the same way). Derivation is deterministic from the row, so no per-blob metadata needs to be stored to recover the key on decrypt — the blob UUID in the `event` row is sufficient. Sibling to the search key Ks in the Section 7 hierarchy. Documented in Section 6's "Encrypted blob storage." Decided rev 30.

- **Thumbnail-first photo storage (M14, rev 30):** Photos are ingested at thumbnail resolution (1024px max edge, ~150KB per photo) rather than full original resolution. Reduces per-user storage by ~95% versus full-resolution (rough estimate: a user with 30k photos at 5MB original avg = 150GB; thumbnails at 150KB avg = 4.5GB). Full-resolution backfill is a future operation when the user explicitly requests it (print-quality year-in-review output, downloading the original from event detail, sharing high-res publicly). All V1 UI use cases — timeline, detail view, collages, year-in-review, on-this-day — work fine at thumbnail resolution. The `event.blob_ref` jsonb shape can hold both thumbnail and full-resolution variants when backfill ships (see 12.3); storing thumbnail-only in M14 doesn't paint M15+ into a corner. Decided rev 30.

- **Local cache with Storage as source-of-truth for blob delivery (M14, rev 30):** First-time view of a blob fetches and decrypts from Supabase Storage; the decrypted bytes are cached in the app's private storage (Android: app-private internal storage; desktop: Tauri app data directory); subsequent views are served from the local cache without network or decrypt cost. Versus the alternative of download-on-view-every-time: better offline behavior (recently-viewed photos remain available without network), lower bandwidth on cellular (especially relevant for Android), faster scrolling through years of photos. Cost: real cache management code — eviction policy (LRU with size cap), invalidation when blobs are deleted server-side, handling of cache corruption, separate caches per user identity to avoid cross-account bleed. The cache holds *decrypted* bytes, so it requires the same OS-level filesystem isolation guarantees the app already relies on for the OS keychain entry holding Kd; no additional encryption layer at the cache level. Decided rev 30.

- **BlobEncryptor instantiation strategy (M14 phase 1 finding, rev 33; prescription corrected rev 44):** Initial `Cipher.getInstance('AES/GCM/NoPadding')` on the JVM has a ~35ms JCE-provider warmup; subsequent calls on the same provider drop to single-digit ms. **Phase 4's `WorkManager`-based ingestion absorbs the `Cipher.getInstance` warmup once per process via the JCE provider cache; explicit per-thread `BlobEncryptor` caching is not implementable (single-use by JCE `Cipher` contract — per-blob keys plus per-blob random nonces require a fresh `Cipher.init()` per blob).** An optional throwaway `Cipher.getInstance('AES/GCM/NoPadding')` call at Worker `doWork()` entry absorbs the warmup before the first photo's timing-sensitive path; the M14 Phase 4 implementation does this. For Phase 2 / Phase 3 single-blob foreground operations (user-initiated "Sync gallery" or single-photo upload from event detail), the warmup cost amortizes against the network round-trip and is acceptable. Surfaced during Galaxy Z Fold 6 throughput profiling on the Kotlin side (rev 33); the Rust side has the same warmup characteristic via `aes-gcm`'s `Aes256Gcm::new` though the absolute cost is lower. The original rev 33 prescription text ("instantiate one `BlobEncryptor` per worker thread upfront and reuse it for the duration of the worker's lifetime, not allocate per-blob") was imprecise — Phase 4 investigation surfaced that `BlobEncryptor` cannot be reused across blobs by construction (single-use by JCE `Cipher` contract), and the warmup amortization that rev 33 was correctly observing comes from the JCE provider cache rather than from explicit instance reuse. The finding stands; the prescription is corrected. Decided rev 33; prescription corrected rev 44.

- **Decryption tag verification timing (M14 phase 1 finding, rev 33):** JDK `Cipher` reports AES-GCM authentication errors **only at `doFinal()`** — `update()` calls return ciphertext-XOR'd-bytes that may fail authentication once the tag is checked at finalize. **Phase 3 photo rendering must buffer all decrypted bytes before displaying — no progressive rendering of in-flight decryption.** A photo that begins rendering at the first `update()` return and then errors at `doFinal()` would expose a partially-rendered tampered or corrupted image to the user, which is worse than a clean failure. For V1's thumbnail-first ingestion (~150KB blobs), full buffering before display is fine; the decode-and-decrypt happens fast enough that no progressive UX is missed. A future full-resolution rendering milestone (or video ingestion) may need to revisit this — true progressive decryption requires a different AEAD construction (per-chunk authenticated; STREAM construction or AES-GCM-SIV-with-chunking) and the Section 12.3 V2 streaming upgrade path becomes the natural carrier. Decided rev 33.

- **Single-shot AES-GCM throughput sufficient for V1 photo workloads (M14 phase 1 finding, rev 33):** Measured ~10 MB/s on Galaxy Z Fold 6 (single-threaded) through the chunked API surface. At the 1024px-max-edge thumbnail sizes specified by Section 12.6's "Thumbnail-first photo storage" decision (~150KB average per photo), per-blob encrypt+decrypt is ~15ms. A 30,000-thumbnail archive ingest is ~7–8 minutes of CPU time before network — and parallelizable across worker threads, so wall-clock is much lower in practice. **Validates option β (single-shot AES-GCM under the chunked API surface) as the right V1 choice — no need to revisit `aes-gcm-stream` or true streaming for thumbnail-first workloads.** Phase 5's hypothetical full-resolution backfill (5MB originals at ~500ms each per the same throughput) would put a 30k-photo archive at ~4.5 hours of single-threaded CPU and would benefit from overnight-scale framing or aggressive parallelism, but that's a separate future milestone (the existing Section 12.3 "Full-resolution photo backfill" follow-up). Decided rev 33.

- **End-to-end encryption forbids server-side content validation (M14 phase 2 finding, rev 36):** Encrypted blobs are opaque ciphertext on the wire; their honest `Content-Type` is always `application/octet-stream` regardless of what they decrypt to. The original `blobs` bucket configuration (rev 34, migration `0006_blob_storage.sql`) included a MIME allowlist for image types (`image/jpeg`, `image/heic`, `image/heif`, `image/png`, `image/webp`) modeled after a non-encrypted bucket. The allowlist blocked 100% of legitimate uploads because the client correctly declares `application/octet-stream` for encrypted bytes — declaring a plaintext content-type would lie about what's stored *and* leak metadata about the original content. The allowlist was removed in migration `0007_blob_storage_remove_mime_allowlist.sql` (rev 36). **General principle:** for end-to-end-encrypted storage, server-side content validation is structurally incompatible with the encryption story. The server can validate **metadata** (size, naming, RLS scope) but cannot validate **content**. Content-shaped checks (MIME validation, dimension limits, malware scanning, moderation, NSFW classification) must happen **client-side before encryption**, or not at all — and "not at all" is a defensible choice because every check has cost and many checks are duplicative once the threat model is "the user trusts their own device and we trust nothing else." The remaining gates on the `blobs` bucket are the meaningful ones: RLS (per-user prefix scoping), `file_size_limit` (50MB ceiling), and `public = false` (no anonymous reads). Decided rev 36.

- **Blob URL transport for decrypted images on desktop (M14 Phase 3, rev 40):** Three options were considered for transporting decrypted JPEG bytes from Rust to the webview: **(a)** blob URL via `URL.createObjectURL(blob)`, **(b)** data URL via base64 encoding (`data:image/jpeg;base64,...`), **(c)** custom Tauri protocol via the `tauri://` scheme. *Decision:* **blob URL.** *Reasons:* conventional React pattern (every React image-loading library uses blob URLs as the canonical `<img src>` value), mature lifecycle (`URL.revokeObjectURL` on component unmount cleanly releases the underlying memory), no 33% base64 inflation that would balloon thumbnail blobs from ~150KB to ~200KB on the wire to the webview, works without friction under the existing `"csp": null` configuration. Custom Tauri protocol is more future-proof for full-resolution images and video (zero-copy from Rust to webview, no buffer duplication) but premature for thumbnail-first M14 (~150KB blobs where the copy cost is invisible). *Forward-compatibility:* the blob URL pattern can coexist with a custom protocol later if full-resolution rendering or video work warrants it — the boundary is the `useEncryptedBlob` hook's return value (`{ url, isLoading, error }`), and the URL string can be either form transparently to consumers. Decided rev 40.

- **Paranoid wipe of local blob cache on signout, both platforms (M14 Phase 3, rev 40):** The local blob cache holds **decrypted bytes** — by definition cleartext personal photos sitting in the app's private storage directory. On signout (data key transitions to null), the cache directory is **recursively deleted** on both desktop (`<app_local_data_dir>/<user_uuid>/blob_cache/`) and Android (`<filesDir>/blob_cache/<user_uuid>/`) rather than left in place. The "lazy mode" alternative (leave bytes; useless without the data key) is structurally true — the bytes are AES-GCM-encrypted on Storage and the decryption key is gone — but reduces the privacy posture from *"signing out wipes your local data"* to *"signing out wipes the keys but leaves cleartext on disk."* The first is what users expect when they sign out of a privacy-first product; the second is exploitable by an attacker with disk access (post-signout, pre-overwrite) and a forensic recovery tool. **Wipe is recursive directory delete** — small implementation cost on both platforms (`std::fs::remove_dir_all` in Rust; `File.deleteRecursively()` in Kotlin). Aligns with the broader privacy story we tell users about end-to-end encryption: keys vanish *and* the cleartext working-copy vanishes. Decided rev 40.

- **Encrypted-blob primitive built directly, no generic remote-image abstraction (M14 Phase 3, rev 40):** The question of whether to build a generic `useNetworkImage` / `RemoteImage` primitive that the encrypted-blob pipeline extends, vs. building the encrypted-blob-specific case directly, was considered. *Decision:* **encrypted-blob directly.** *Reasons:* only one current use case (encrypted blobs from the M14 pipeline); Spotify album-art rendering is a hypothetical second use case but minor (and arguably should be ingested as event-level metadata, not loaded ad-hoc per render); the encrypted path has more concerns than a generic remote image (per-blob HKDF key derivation, AES-GCM decryption, paranoid-wipe on signout, cache-directory scoping per user) — a premature generic abstraction would force the unencrypted case to carry encrypted-specific complexity in its API even when it doesn't need it. **YAGNI.** If Spotify album art (or any other generic remote image) ships as a meaningful design surface later, extracting a shared primitive at that point earns its weight against a real second consumer. For now: **`useEncryptedBlob(blob_ref)` hook on desktop** returning `{ url, isLoading, error }`; **`EncryptedBlobImage` composable on Android** wrapping `AsyncImage` + `Fetcher.Factory<BlobRef>`. Decided rev 40.

- **Decryption inside TanStack Query `queryFn`, not in downstream `useEffect` (M14 Phase 3 desktop, rev 40; refactored M15 closing, rev 47):** The original `apps/desktop/src/components/timeline/timeline.tsx` pattern decrypted in a downstream `useEffect` that watched the `useInfiniteQuery` result and populated a separate `useState<DecryptedEvent[][]>`. This pattern has two costs: it re-runs decryption on every parent re-render that doesn't re-fetch, and it breaks `isLoading` semantics (the component reads "fetched but not yet displayable" as if it were "ready," because TanStack's `isLoading` only reflects the network fetch). M14 Phase 3 set a better precedent: the `queryFn` (or the `select` option) returns the **final displayable shape** — a decrypted blob URL string — so `isLoading` correctly reflects "ready to display" and re-renders don't trigger re-decryption. The existing `timeline.tsx` pattern was left as-is at rev 40 (not breaking anything in production at the M14 Phase 3 scale of ~2k events; the rewrite was captured as a side-quest "implicitly by the duplicate-`QueryClientProvider` § 12.3 follow-up's 'next desktop refactor touches App.tsx' trigger"), while **Phase 3's `useEncryptedBlob` hook did it right from day one** so the better pattern was established alongside the existing one. Decided rev 40. **Update (rev 47, M15 closing):** the "stays as-is for now" framing held until M15 real-archive verification — 10k YouTube watch-history events on top of 1.3k Android gallery photos plus ~100 misc (11.4k total) surfaced the predicted quadratic cost from the `useEffect`-rebuilds-on-every-page-arrival pattern. The `useEffect` iterated `data.pages` from scratch every time a new page arrived from `useInfiniteQuery`, re-decrypting all previously-loaded rows; cumulative IPC count to scroll a 228-page timeline at `PAGE_SIZE = 50` is 50·∑1..228 ≈ 1.3 million decryption calls (the same row at page index *i* is re-decrypted *N − i* times by the time the user reaches the bottom). At the M14 ~2k-event scale (40 pages) this was ~41k cumulative IPC calls — slow but tolerable; at M15 scale it blocked the UI thread. `apps/desktop/src/components/timeline/timeline.tsx` refactored in M15 closing to put decryption inside the `queryFn` per the rev 40 prescription; the `useEffect` and the `decryptedPages` `useState` are deleted, `queryFn` returns `DecryptedEvent[]` directly, and TanStack now caches decrypted pages by page-cursor key so an already-fetched page is never re-decrypted. The better pattern (`useEncryptedBlob`) and the worse pattern (timeline `useEffect`) no longer coexist; the timeline is the canonical example for any future `useInfiniteQuery + per-row decrypt` pipeline. *Rationale for the doc correction:* rev 40 anticipated this exact tipping point ("not breaking anything in production" implied "until production scale grows enough to break it"); M15 was the milestone where scale finally caught up.

- **Coil 3.4.0 for Android image rendering (M14 Phase 3, rev 40):** Phase 3 introduces image rendering on Android for the first time. Coil was documented as the intended library in `apps/android/.cursorrules` since project inception but was never actually added to `gradle/libs.versions.toml` or `app/build.gradle.kts` — the file recorded an intent against a future need. *Decision:* pin **Coil 3.4.0** (current stable as of investigation). *Rationale:* Coil 2.x is in maintenance — 2.7.0 (July 2024) is the final 2.x release, no further 2.x work planned; migration from 2 → 3 later is real work (package rename `coil` → `coil3`, Maven coordinate change `io.coil-kt` → `io.coil-kt.coil3`, `SourceResult` → `SourceFetchResult` rename, plus subtler renames in the network-fetcher artifact split); Coil 3.4+ adds `memoryCacheMaxSizePercentWhileInBackground` for backgrounded memory pressure (relevant for typical phone usage where the founder may leave the app backgrounded for hours or days at a stretch); the multiplatform-ready abstractions (`coil3.Image`, `coil3.Bitmap`) cost nothing on Android and become free upside if any future iOS work materializes. The `Fetcher.Factory<T>` / `Keyer<T>` / `DiskCache` / `MemoryCache` API surface is **substantively the same** between 2.7 and 3.x — only renames differ — so all Phase 3 design decisions hold for either version, and starting on 3 avoids a mandatory migration later. `apps/android/.cursorrules` is updated in this rev to specify the version and the fetcher pattern. Decided rev 40.

- **Android blob cache directory uses `filesDir`, not `cacheDir` (M14 Phase 3, rev 40):** Coil's official documentation example uses `context.cacheDir.resolve("image_cache")` as the disk cache directory. Phase 3 deviates: uses `context.filesDir.resolve("blob_cache/<user_uuid>")`. *Rationale:* `cacheDir` is the Android system's "wipeable under storage pressure" directory — the OS may silently reclaim contents when storage runs low, with no app notification. Our cache holds **decrypted bytes** that are CPU-expensive to regenerate (Supabase Storage round-trip + Tink AES-GCM decrypt + JPEG decode per blob). Losing the cache silently to OS pressure means re-paying that cost across thousands of thumbnails on next scroll. `filesDir` is the "survives until app uninstall or explicit *Clear data*" directory, which is the durability profile our working-copy-of-archive needs. **Per-user-prefix subdirectory** (`blob_cache/<user_uuid>/`) for tenant isolation on shared devices — mirrors the rev 34 Storage path convention so the local cache layout matches the remote layout. Recursive directory deletion on signout per the paranoid-wipe decision above. *Cost:* `filesDir` storage counts against the app's "App size" rather than "Cache size" in Android Settings, which is a slightly less convenient story for the founder ("why is Flashback using 500MB?") — mitigated by surfacing cache size in the desktop+mobile settings and offering a manual "Clear local cache" button as a future Phase-3.5 nice-to-have. Decided rev 40.

- **Coil 3.4 does not auto-tee `Buffer`-backed `SourceFetchResult` to disk for custom fetchers (M14 Phase 3 implementation finding, rev 42):** The rev 40 Coil decision assumed (per Coil's documentation example with `HttpUriFetcher`) that returning a `SourceFetchResult` from a custom `Fetcher` would automatically populate the disk cache. Verified empirically on the Galaxy Z Fold 6 during M14 Phase 3 verification: the cache directory stayed empty across multiple fetch cycles even though `EncryptedBlobFetcher.fetch()` was returning valid `SourceFetchResult` instances. *Fix:* explicit `DiskCache.openSnapshot` / `openEditor` / `commitAndOpenSnapshot` inside the custom `Fetcher`, manually writing the decrypted bytes to the data slot. The auto-tee behavior is specific to Coil's built-in network fetchers (which use `openSnapshot` internally via `NetworkFetcher`) and does not extend to user-supplied `Fetcher` implementations even when they emit the same `SourceFetchResult` shape. Decided rev 42 (M14 phase 3 implementation).

- **Coil 3.4 `DiskCache` requires non-empty metadata slot for cold-start reopen validity (M14 Phase 3 implementation finding, rev 42):** `DiskCache` entries have two slots — metadata (slot 0) and data (slot 1). Coil's built-in `NetworkFetcher` writes both: HTTP response metadata to slot 0, body bytes to slot 1. A custom fetcher writing only the data slot produces an entry that survives the in-process session (the in-memory journal accepts it) but `openSnapshot` returns null on cold-start journal replay — the entry is treated as invalid despite the journal listing it as `CLEAN`. *Fix:* write a 1-byte sentinel value to `editor.metadata` before commit. Required for any cross-session cache hit; otherwise force-stop + reopen invalidates all cache entries and every photo re-fetches. Surfaced during M14 phase 3 verification on Galaxy Z Fold 6. Decided rev 42 (M14 phase 3 implementation).

- **WorkManager 2.10 default startup init runs before `Application.onCreate()` and uses default `Configuration` (M14 Phase 4 implementation finding, rev 44):** WorkManager auto-initializes via `androidx.startup.WorkManagerInitializer` at `ContentProvider` startup, **before** `Application.onCreate()`. The `Configuration.Provider` interface implemented on the `Application` class is never queried because WorkManager is already initialized with the default `Configuration` by the time `Application.onCreate()` runs. Surfaced during M14 Phase 4 verification on Galaxy Z Fold 6: the first "Sync now" tap failed with `WM-WorkerFactory` falling back to reflection (`"Could not instantiate GallerySyncWorker / NoSuchMethodException: <init> [Context, WorkerParameters]"`) because the custom `CodexWorkerFactory` wasn't being used. *Fix:* opt out of auto-init in `AndroidManifest.xml` via the `InitializationProvider` `tools:node="remove"` merge:

    ```xml
    <provider
        android:name="androidx.startup.InitializationProvider"
        android:authorities="${applicationId}.androidx-startup"
        android:exported="false"
        tools:node="merge">
        <meta-data
            android:name="androidx.work.WorkManagerInitializer"
            android:value="androidx.startup"
            tools:node="remove" />
    </provider>
    ```

    With this opt-out, WorkManager initializes on-demand at the first `getInstance()` call, at which point `Configuration.Provider` is queried and the custom `WorkerFactory` is honored. This is a well-documented but easy-to-miss step in the WorkManager custom-configuration guide — the `Configuration.Provider` interface alone is not sufficient; the manifest opt-out is the load-bearing piece that makes the interface contract take effect. Decided rev 44 (M14 Phase 4 implementation).

- **Takeout parsers are TypeScript-first; escalate to Rust per-category when memory or perf demands it (M15 architectural decision, rev 46):** A parser escalates from TS to Rust when **any** of the following holds: (a) the source file exceeds ~200 MB raw and benefits from streaming-parse (Rust + `serde_json::Deserializer::from_reader` keeps a fixed-size working set while V8 would slurp the whole thing); (b) the parser does CPU-heavy normalization (image decode, geo coordinate projection at scale) that V8 would block the UI thread on; (c) the category needs sustained binary handling that's awkward in TS (KML/KMZ extraction with embedded compressed payloads). *Default is TS* because `packages/parsers/` is already a TS pnpm workspace package (the existing scaffold has WhatsApp + Instagram stub files in TS), JSON parsing is trivial in V8 at the working sizes the Tier 1 Takeout categories produce, and the ingestion-orchestrator pattern in `apps/desktop/src/lib/ingestion/google.ts` is already TS-side. V1 M15 ships only TS parsers (YouTube watch history at ~5–50 MB `watch-history.json` is comfortably medium and parses to ~150–200 MB of objects, well within Tauri's V8 heap ceiling of ~1–2 GB). Maps Timeline at 100 MB+ `Records.json` for multi-year power users is the first plausible escalation candidate at M16 but is *not* pre-decided — M16 picks per the rule above based on actual measurement against a real archive. Decided rev 46 (M15 investigation findings).

- **Takeout parsers derive `source_event_id` via stable hash of source-entry fields (M15 architectural decision, rev 46):** Unlike OAuth-API-derived events (Google provides raw IDs in the API response — see `RawGmailEvent.id`, `RawPhotoEvent.id`, `RawCalendarEvent.id`, etc.) or content-provider-derived events (Android provides content IDs from the underlying `MediaStore` / SMS / Calendar providers), Takeout has **no platform-provided per-entry identifier**. Every Takeout parser MUST derive `source_event_id` such that the same source entry across re-exports produces the same value. The unique constraint `(user_id, source, source_event_id)` on `event` then makes re-uploading idempotent via the established `onConflict: 'user_id,source,source_event_id', ignoreDuplicates: true` Google pattern from `apps/desktop/src/lib/ingestion/google.ts` — zero new code in the orchestrator. *First parser (M15):* YouTube watch history uses `sha256_b64url(entry.titleUrl ‖ '|' ‖ entry.time)`. `titleUrl` is the stable video identifier (`https://www.youtube.com/watch?v=<videoId>`); `time` is the ISO 8601 watch timestamp at second precision (a single video watched twice produces two entries with different `time` values, correctly disambiguated). Localized strings (`header`, `title`) are deliberately excluded because they vary across re-exports as account language changes or videos are re-titled by their creators. *Edge case:* entries for deleted / private videos lack `titleUrl` and present as `title: "Watched a video that has been removed"`; for those, parsers use `sha256_b64url('removed' ‖ '|' ‖ entry.time)` — the loss of identity is intrinsic to the source, not a parser bug. *YouTube Music vs YouTube proper:* `watch-history.json` mixes both; the `header` field disambiguates (`"YouTube"` vs `"YouTube Music"`) and parsers set `event_type = 'listen'` for Music entries and `'watch'` for video entries, but the dedup-key derivation is identical. Future Takeout parsers (Maps Timeline at M16, etc.) pick their own stable-field set per category and document the derivation alongside the parser; the invariant — *same source entry across re-exports → same `source_event_id`* — holds for every parser. Decided rev 46 (M15 investigation findings).

- **Maps Timeline via device-side Android export (M16 scoping decision, rev 48):** Google's December 2024 cloud-side Location History migration moved Timeline data on-device only — for migrated accounts, Takeout exports under "Location History (Timeline)" return only an `Encrypted Backups.txt` placeholder with no JSON; for non-migrated accounts, the cloud-side data was deleted at the December 1, 2024 deadline. Section 11's original "Maps Timeline via Takeout" mini-spec is operationally dead for every user from December 1, 2024 forward. **M16 pivots to device-side export ingestion** via the Android Maps app's *Settings → Location → Location Services → Timeline → Export Timeline data* feature, which produces a single bare JSON file (typically `Timeline.json`, ~5–100 MB depending on retention window — the founder's three-month archive at the device's `retentionWindowDays: 90` default is 22 MB / 908,234 lines) with three top-level keys: `semanticSegments` (the semantic timeline — visit, activity, timelinePath, and timelineMemory entries), `rawSignals` (raw GPS positions, activity records, Wi-Fi scans), and `userLocationProfile` (derived analytics like frequent-place HOME/WORK labels). **V1 ingests `visit` and `activity` entries from `semanticSegments`** — 713 events from the founder's archive (382 visits + 331 activities, distributed 60%/24%/7%/5%/4% across visit `semanticType` values `UNKNOWN` / `HOME` / `INFERRED_WORK` / `SEARCHED_ADDRESS` / `WORK`; activity types are 329 `IN_PASSENGER_VEHICLE` and 2 `WALKING`). **V1 emits two distinct `event_type` values** — `'visit'` (place-and-duration shape) and `'movement'` (activity-with-distance shape) — under one shared `source = 'maps_timeline'`, with two distinct payload types `MapsTimelineVisitPayload` and `MapsTimelineMovementPayload` in `packages/shared/src/schema/maps.ts`. The two-event-type split honours the M14 Phase 3 per-source-card discipline (one event-type per card component, no in-card switching on payload shape) — visits and activities have semantically different rendering needs (place name + duration + map-pin icon for visits; activity-type icon + distance + direction-arrow for movements) and forcing a single `event_type = 'location'` would push the dispatch into the card itself, which is the anti-pattern the per-source-card discipline avoids. **Plugs into M15's `takeoutRegistry` pattern** via an `ingestArchive(unpackedRoot, sourceLabel, dataKey)` helper extracted from `ingestTakeout`: the bare-JSON upload affordance copies the user-selected `Timeline.json` into a fresh `codex-maps-` temp directory (via a new `prepare_maps_workspace` Tauri command — single-file copy, no zip unpack needed) and invokes the same per-parser dispatch loop, so the Maps parser registers with the M15 registry exactly like the YouTube parser. The upload UX is a new `MapsSourceCard` distinct from the Takeout `.zip` uploader (separate "Upload Timeline JSON" affordance with a `.json` file filter), because the input shape is structurally different — a single bare JSON file rather than an archive needing unpack — and the user mental model is "your Maps export" rather than "your Takeout zip." **`source_event_id` derivation** per the rev 46 invariant: visits hash `sha256_b64url(place_id ‖ '|' ‖ start_time)` — every visit in the founder's archive has `topCandidate.placeId` (382 of 382, no `null`s), so no fallback recipe is required at M16; activities hash `sha256_b64url(activity_type ‖ '|' ‖ start_latlng ‖ '|' ‖ start_time)` — activities have no platform identifier, but the `(type, start_latlng, start_time)` triple is stable across re-exports because `start_latlng` is the GPS-derived coordinate string Google emits (e.g. `"30.57°, -88.13°"`) rather than a user-edited field that could drift between exports. **V1.5 follow-ups** captured in § 12.3: `timelinePath` ingestion (670 of 1,386 segments — 48% — in the founder's archive, but text-only rendering can't surface them meaningfully); `timelineMemory` trip summaries (3 entries — too sparse for V1 utility); `userLocationProfile.frequentPlaces` metadata for Sources card display; legacy pre-2024 Takeout format support if non-founder users surface preserved archives containing `Records.json` or `Semantic Location History/<year>_<MONTH>.json`. **Never V1:** `rawSignals` ingestion — Wi-Fi scan data inside `rawSignals.wifiScan.accessPoints[]` contains MAC addresses and signal strength of nearby APs at each location sample, which is uniquely de-anonymizing (an attacker with the same APs in scan range can prove the user was physically nearby), redundant with the semantic visit/activity events the parser does ingest, and inconsistent with the encryption-first privacy story. The temp-directory paranoid-wipe (M15 § 12.6) is already recursive via `std::fs::remove_dir_all` so the deeper directory structure that legacy Takeout's `Semantic Location History/<year>/<year>_<MONTH>.json` would have required is handled with zero code change; the device-side export's flat single-file shape is even simpler. Surfaced during M16 investigation when the founder's actual Z Fold 6 export was inspected (22 MB `Timeline.json`, 908,234 lines, 1,386 `semanticSegments` distributed 670 `timelinePath` / 382 `visit` / 331 `activity` / 3 `timelineMemory`, three-month retention window 2026-02-11 to 2026-05-12 matching the device's `retentionWindowDays: 90` Settings.json value). Decided rev 48 (M16 scoping investigation).

- **Blob URLs created and revoked inside the same `useEffect` for StrictMode safety (M16 closing finding, rev 49):** Blob URLs must be created via `URL.createObjectURL` and revoked via `URL.revokeObjectURL` **inside the same `useEffect`**, with the URL closure-captured as a local variable. The earlier shape used in `apps/desktop/src/hooks/useEncryptedBlob.ts` from M14 Phase 3 through M15+M16 — `useMemo` creates the URL and a paired `useEffect` cleanup keyed on `[url]` revokes it — races fatally under React 18 `<StrictMode>` and is the **incorrect** pattern despite reading as natural. *The race:* (a) StrictMode renders components twice on initial mount as a purity check, so the `useMemo` factory runs twice and registers two `URL.createObjectURL` returns — the first is orphaned forever (no `useEffect` cleanup sees it because the dep `[url]` only watches the *committed* URL, which is the second one), the second becomes the committed `url` state; (b) StrictMode then invokes the effect-mount → simulated-unmount → simulated-remount cycle synchronously after commit, and the simulated unmount's cleanup calls `URL.revokeObjectURL` on the committed URL **before** the browser finishes loading from `<img src>`. The browser then 404s the dead `blob:` URL with `net::ERR_FILE_NOT_FOUND` and the `<img>` renders as a broken-image icon. Symptom: every photo, on every fresh mount of an `AndroidGalleryPhotoCard`. *Why the pattern looks correct on inspection:* the `useMemo` deps `[query.data, contentType]` are stable across re-renders (the TanStack-cached `Uint8Array` is the same reference; the `contentType` string is value-equal), so once mounted the URL should never churn — and indeed under non-StrictMode production it doesn't. The bug is exclusive to the synthetic-double-invoke that StrictMode does to surface impurity, and it bites only on the *initial mount* of a fresh component instance (subsequent re-renders are stable). This is what made the bug latent through M14 Phase 3 verification (small archive; many photos rendered correctly because HMR re-mounts during dev verification happened to bypass the synthetic-unmount path enough times) and surface consistently after M15 closing's rev 47 `timeline.tsx` `queryFn`-decrypt refactor changed initial-mount render-cycle timing in a way that consistently exposes the race. *The correct pattern* — refactored in this rev — is `useState<string | null>` for the URL plus a single `useEffect` keyed on the byte-payload deps that calls `URL.createObjectURL` AND `URL.revokeObjectURL` against a closure-captured local: each effect invocation owns its own URL, the synthetic-unmount path revokes the first invocation's URL cleanly, the synthetic-remount path creates a fresh URL that is still alive when React batches the two `setUrl` calls and commits the final render, and there is no shared mutable URL state that the cleanup can prematurely invalidate. *Generalization for future hook authors:* **any hook creating a browser-resource handle that needs explicit teardown — object URLs (`URL.createObjectURL`), audio contexts (`new AudioContext()`), `MediaStream`s, `RTCPeerConnection`s, `IntersectionObserver`s, `ResizeObserver`s, `MutationObserver`s, `EventSource`s, `WebSocket`s — must use the in-same-effect pattern.** The `useMemo`-creates + `useEffect`-revokes split is a known anti-pattern under React 18 StrictMode and looks fine on inspection until the synthetic double-invoke bites in dev. The inline comment in `useEncryptedBlob.ts` documents the rule for the next reader. *Side benefit:* the orphaned-URL leak that StrictMode's `useMemo` double-invoke produced (every fresh mount leaked one URL in dev mode — production was unaffected because StrictMode is dev-only, but dev-mode leaks compound across HMR cycles and bloat memory during a long debugging session) is also resolved by the fix as a free byproduct. Decided rev 49 (M16 closing finding).

- **M10 verification scaffolding bugs (evidence for rules 12 and 13):** Real verification — Android Studio Gradle resolve, Kotlin compile, install on physical hardware, sign in, decrypt a live event — caught seven scaffolding bugs that static review and pre-build inspection didn't: (1) Supabase Kotlin SDK pinned at the non-existent `2.7.4` instead of the actual latest 2.x `2.6.1`; (2) manifest referenced an invented `Theme.Material3.DynamicColors.DayNight` that doesn't exist in any shipped Material Components release; (3) `com.google.android.material:material` dependency missing from the Gradle catalog despite the system theme reference requiring it; (4) all six Android `.gitignore` patterns rooted at `android/...` against the actual `apps/android/...` layout — `local.properties` with real Supabase credentials was un-ignored, one `git add .` away from a credential leak; (5) GoTrue plugin renamed `GoTrue` → `Auth` between the SDK version Cursor coded against and 2.6.1, breaking the `install(GoTrue)` call site; (6) Gradle wrapper generated by Android Studio but never `git add`ed — a "works on my machine" landmine for any CI run or future contributor; (7) test-vectors README's `./gradlew :app:test --tests "..."` rejected because the lifecycle `:app:test` task doesn't accept `--tests` (variant `:app:testDebugUnitTest` does). Pattern across all seven: rule 12 (`pnpm typecheck` before declaring complete) generalizes to every platform's analogous build+test gate, and rule 13 (verify `.gitignore` patterns actually match the paths they intend to catch) is one specific instance of "verify the assertion actually holds rather than assuming it." Both rules earned their place by failing for real on this milestone.

## 13. Working with Cursor on this codebase

If you are an AI coding assistant, follow these rules:

1. **Read this document fully before any architectural change.**
2. **Do not invent features, dependencies, or schema fields.** If something seems missing, ask the user.
3. **Match the established style.** When adding code, follow patterns in adjacent code. Prefer fewer files over more files. Co-locate related code.
4. **Encrypt-by-default.** Any new field or table holding user content must be encrypted client-side before reaching the server. Adding plaintext content fields to the schema is a breach of architecture and requires explicit user approval.
5. **Use the existing crypto helpers.** Do not roll your own crypto. Use `packages/shared/src/crypto/` exclusively.
6. **Do not add telemetry or analytics that report user content.** Anonymous usage analytics (page views, feature usage) are acceptable later; user content metrics are not.
7. **Tests for parsers, crypto, and schema migrations are required.** Tests for UI components are nice-to-have.
8. **Commit messages: imperative mood, short subject + body.** Reference this document when making architectural decisions.
9. **When in doubt, smaller scope is better.** This is a V1 shipping to one user (the founder) first.
10. **Sections 5 and 10 must stay in sync.** When adding or removing an Edge Function, update both the Section 5 directory listing and the Section 10 API table in the same change.
11. **Build artifacts are never committed.** This includes `node_modules/`, `target/`, `dist/`, `build/`, `.next/`, `apps/desktop/src-tauri/gen/`, and any other generated directory. The `.gitignore` is the source of truth for what's excluded; if a generated file is being committed, the `.gitignore` is wrong, not the gitignore policy.
12. **Before declaring any milestone complete, run `pnpm typecheck` (or whatever the project's typecheck command is) and verify it passes with zero errors.** M8 shipped with two pre-existing typecheck errors that weren't caught until M9 verification needed a clean baseline.
13. **Verify gitignore patterns match actual file paths, not assumed paths.** Use `git check-ignore -v <path>` to confirm a pattern is actually catching a file. Files containing secrets (`.env`, `local.properties`, etc.) must be verified to be ignored, not just observed as untracked. Untracked files are one `git add .` away from accidental commit. Triggered when M10 verification found six Android `.gitignore` patterns rooted at `android/...` that silently no-op'd against the actual `apps/android/...` layout — `local.properties` with real Supabase credentials was relying on nobody running `git add .` from the repo root.
14. **When switching between AI agents (Cursor, Claude Code, etc.), the new agent's first action must be `git pull` to sync local state with origin.** Don't assume your local working tree reflects the latest commits. M10 verification surfaced multi-agent coordination as a real failure mode; this rule makes the handoff explicit.
15. **RLS UPDATE policies require both `using` and `with check` clauses.** The `using` clause filters which existing rows the statement can target; the `with check` clause filters what the row can become post-update. Without `with check` on an UPDATE policy, an authenticated user can rewrite their row into another user's tenant scope — the original path matches `auth.uid()` so `using` permits the update, but no constraint guards the new path. Surfaced during M14 Phase 2 storage-bucket setup (rev 34, `storage.objects` UPDATE policy `blobs_user_update`); applies equally to any future RLS UPDATE policy on `public.*` tables or other `storage.*` buckets.
16. **When querying PostgREST endpoints, always specify explicit `limit` and pagination behavior.** PostgREST applies a default `X-Max-Rows` of 1000 and **silently truncates** rather than erroring when the result set exceeds it. For dedup queries, source-event-id lookups, content-hash existence checks, or any query whose correctness depends on returning *all* matching rows, pagination through the full result set (range headers or cursor-based) is required regardless of expected size at write time. "It will only ever be a few hundred rows" is not a safety margin — a year of photo ingestion or a single Takeout import easily clears 1000. Surfaced during M14 Phase 2: a 1000-row truncation on a 1351-event archive caused 351 photos to appear "missing" on re-sync, triggered re-upload of those 351 thumbnails to Storage, and produced the orphan-blob accumulation tracked in Section 12.3 ("Orphan blob cleanup"). The truncation is silent; the only signal at runtime is "wrong answer," not an error or warning.
